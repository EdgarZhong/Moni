# Bill Import UI/UX Design Brief

## 状态

accepted for implementation

## 背景

账单导入后端已经完成增强，当前支持直传文本 / CSV、Excel、加密压缩包探测，并收口为“先 `probe`，再 `import`”的接口模式。

当前记账页已有“微信账单”和“支付宝账单”两个导入按钮。用户已确认：本轮不重做记账页表面，不新增导入舱，不改变两个按钮入口结构。UI/UX 设计重点应放在两个按钮点击后的最小状态流：文件选择、探测反馈、按需密码输入、导入反馈与失败恢复。

## 目标

- 保留记账页现有“微信账单 / 支付宝账单”两个按钮入口。
- 明确两个按钮后续行为差异仅为 `expectedSource` 与平台化密码文案。
- 设计一个低摩擦、可落地的导入状态流，让用户理解“正在识别 / 需要密码 / 正在导入 / 导入成功 / 无法导入”。
- 避免用户在不需要密码的 CSV / Excel / 明文账单场景中看到多余输入。
- 为正式实现提供可直接对齐 `probeBillImportFiles()` 和 `importBillFiles()` 的 UI 状态定义。

## 非目标

- 不重做记账页导入卡片的视觉结构。
- 不新增统一“导入舱”、文件识别大面板或多步骤全屏向导。
- 不把微信和支付宝拆成两套前端解析逻辑。
- 不要求前端自行判断文件类型、压缩包结构或账单来源。
- 不设计导出账单教程页；现有“查看导入指南”入口如需改造，另开 brief。
- 不在本 brief 中处理 Android 文件选择器底层兼容问题；这里只定义用户可见状态。

## 用户场景

- 用户点击“微信账单”，选择微信导出的普通账单文件，系统识别后直接导入。
- 用户点击“微信账单”，选择微信加密压缩包，系统提示输入微信账单压缩包密码，密码正确后导入。
- 用户点击“支付宝账单”，选择支付宝加密压缩包，系统提示输入支付宝账单压缩包密码，密码正确后导入。
- 用户输入错误密码，系统提示对应平台密码不正确，并允许原地重试。
- 用户选择了不支持或无法解析的文件，系统展示后端返回原因，并允许重新选择。
- 用户导入成功后，系统以轻量反馈告知导入条数和来源平台，不打断继续随手记。

## 涉及页面

- 记账页 `MoniEntry`

## 涉及组件

- 现有账单导入卡片
- 现有“微信账单 / 支付宝账单”按钮
- 账单密码二级页
- 导入状态反馈 Toast / Notice

## 状态覆盖

- 默认态：两个按钮保持现有入口形态。
- 选文件态：点击按钮后打开系统文件选择器，前端记录当前来源。
- 探测中：选择文件后提示“正在识别微信账单”或“正在识别支付宝账单”。
- 密码缺失态：`password_required + missing` 时进入对应平台账单密码二级页。
- 密码错误态：`password_required + invalid` 时保留在二级页并提示密码不正确。
- 可导入态：`ready` 时直接进入导入，不要求用户二次确认。
- 导入中：密码正确后立即回到记账页，在导入卡片底部提示“正在导入微信账单”或“正在导入支付宝账单”。
- 成功态：仍停留在记账页导入卡片区域，展示导入条数，可附带“来自微信 / 支付宝”，不进入独立成功页。
- 不支持态：`unsupported` 时展示后端 `message`，并提供重新选择。
- 取消态：用户关闭文件选择器或取消密码输入后，回到默认态，不保留错误提示。

## 原型需求

需要 prototype，但范围必须小：

- 只展示记账页导入卡片附近的局部区域。
- 保留两个按钮入口，不重建整页。
- 展示以下状态切片：
  - 默认态
  - 微信压缩包需要密码
  - 支付宝密码错误
- 回到记账页后的导入中提示
- 回到记账页后的成功反馈
  - 不支持文件反馈
- 使用 mock 数据，不连接真实后端。
- 通过独立路径 `/__design/bill-import-uiux` 预览。
- 原型必须可交互，不能只做静态状态截图。
- 原型需要直接原样引用正式 `MoniEntry` 页面，不改写正式页面内部实现，也不重写一份记账页。
- 密码输入采用二级页，不在导入卡片内联展开。

## 初步设计方向

本轮建议采用“现有按钮入口 + 密码二级页”的方案。

入口仍是现有两个按钮：

- 微信账单：点击后设置 `expectedSource = "wechat"`。
- 支付宝账单：点击后设置 `expectedSource = "alipay"`。

选择文件后，前端先执行探测。无需密码时可直接进入导入反馈；需要密码时跳转到导入密码二级页，避免在导入卡片内联展开后挤压下方随手记区域。

密码页建议包含：

- 平台标题：微信账单密码 / 支付宝账单密码。
- 辅助说明：输入导出压缩包里的 6 位数字密码。
- 6 位数字密码输入格。
- 使用系统数字键盘能力，不在页面内自定义数字键盘。
- 返回入口。
- 错误提示：密码不正确，请重新输入。

成功反馈建议保持轻量：

- “已导入 73 条微信账单”
- “已导入 37 条支付宝账单”
- 解析中、导入中与成功提示优先占用导入卡片底部提示条；默认“查看导入指南”提示在这些状态下让位。

失败反馈不自行改写后端原因：

- 使用后端 `message` 作为主说明。
- 附加操作：“重新选择文件”。

## 待拍板问题

- 密码输入是否最终采用二级页：当前原型倾向采用二级页，但导入中和成功反馈回到记账页卡片内完成。
- 导入成功反馈是否只保留 3 秒 Notice，还是保留到用户下一次操作？
- `ready` 后是否直接导入，还是展示一次“已识别 N 条，开始导入”的确认？当前建议直接导入，减少摩擦。
- 不支持文件时是否展示文件名摘要？当前建议只展示后端 `message`，避免信息过载。
- 同一次选择多个文件时，成功文案是否展示总条数即可？当前建议只展示总条数。

## 拍板结论

- 用户已认可当前设计，暂停继续围绕 prototype 工作流做探索，本轮直接进入主仓库正式实现。
- 入口保持现有“微信账单 / 支付宝账单”两个按钮。
- 微信 / 支付宝只区分 `expectedSource` 与平台化密码文案。
- 压缩包密码采用二级页输入，导入中与成功反馈回到记账页导入卡片底部提示条完成。
