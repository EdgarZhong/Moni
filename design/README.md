# Moni Design Workspace

`/design` 是 Moni 主仓库唯一的设计工作台，用于承接所有 UI/UX 变更在正式编码前的讨论、原型、审查、拍板与稳定基线沉淀。

## 设计定位

- `/design` 是所有 UI/UX 改动的唯一设计入口。
- `/design` 是正式实现前的讨论区、原型区、审查区与拍板区。
- `/design` 是编码实现时的单向参考源，不直接承载运行时代码。
- `/design` 是设计与代码保持一致的前置约束来源；正式代码必须对齐已拍板 design 资产。
- 原型探索固定分为两种模式：`镜像探索模式` 与 `沙箱探索模式`。
- `mirror / sandbox` 只描述“原型怎么做”，不描述“是否可以改正式代码”。
- 是否进入正式实现，是独立于 `mirror / sandbox` 的另一道阶段门；未明确进入“阶段 6：正式实现”前，默认仍处于设计阶段。

## 根目录组织

```text
design/
├── README.md
├── briefs/
├── brand/
├── components/
├── flows/
├── standards/
├── prototypes/
└── decisions/
```

各目录职责：

- `briefs/`：设计讨论区。所有 UI/UX 任务必须先从 `briefs/active/` 开始。
- `brand/`：品牌级稳定规则，如颜色、字体、Logo、Memphis 风格。
- `components/`：组件级稳定规则，一个组件一个文件，只记录已确认规范。
- `flows/`：流程级稳定规则，一个流程一个文件，描述任务路径、状态变化、异常边界。
- `standards/`：跨组件、跨流程的执行标准，如移动端触控、Pointer Events、WebView、安全区、原型展示协议。
- `prototypes/`：局部 React + TypeScript 原型代码区，只服务设计审查。
- `decisions/`：Design Decision Record，记录重要设计取舍。

## 工作流

### 阶段 1：Design Brief

1. 在 `design/briefs/active/{feature}.md` 创建 brief。
2. brief 必须写清：
   - 背景
   - 目标
   - 非目标
   - 用户场景
   - 涉及页面
   - 涉及组件
   - 状态覆盖
   - 原型模式：`mirror / sandbox / none`
   - 选择该模式的原因
   - 设计约束
   - 是否需要 prototype
   - 待拍板问题
3. 该阶段允许不稳定、允许讨论、允许多方案并存。

### 阶段 2：设计探索

1. 围绕 active brief 讨论收口。
2. 若只是文案、小规则、小边界调整，可只修改 brief。
3. 若需要视觉判断、交互判断、布局判断，进入 prototype 阶段。
4. 在结论未稳定前，不得提前污染 `components/`、`flows/`、`brand/`、`standards/`。

### 阶段 3：局部可视化原型

通用前提：

1. 在 `design/prototypes/{feature}/` 建立局部原型。
2. 原型统一使用 React + TypeScript。
3. 原型必须能通过开发态独立 URL 预览，路径固定为 `http://localhost:{port}/__design/{feature}`。
4. `__design` 根路径只允许作为索引页，不得把多个原型直接串联渲染在同一页面。
5. 原型 UI 只展示用户真实可见文案，不得把设计决策原因、实现解释、评审说明写进界面正文。
6. 移动端输入必须优先使用系统输入法能力，例如 `inputMode="numeric"`、`pattern="[0-9]*"`、`autocomplete`；除非产品明确要求，不得手搓数字键盘替代系统键盘。
7. 原型交互必须贴近真实用户路径：能在原页面轻量反馈的问题，不额外制造确认页、成功页或等待页；只有真正需要独立上下文的任务才进入二级页。
8. 若有多个方案，可在同一 feature 下保留 variant，并通过 `__design/{feature}/{variant}` 或页面内明确切换隔离。
9. 处于原型阶段时，默认交付物只有 `brief / prototype / 审查结论 / 设计基线文档`，不包含正式运行时代码修改。

#### 镜像探索模式 `mirror`

定义：

- 直接原样引用正式页面或正式组件，在其上叠加只存在于 `__design` 的开发态壳层、覆盖层、交互脚本或测试夹具。
- 目标是高保真、低污染、低浪费地验证已有正式实现上的小范围设计问题。

适用场景：

- 在稳定页面上增补一个新小功能点。
- 微调已有交互、状态、文案、提示位置、反馈时机。
- 验证某个局部状态是否会挤压现有布局或打断现有主流程。
- 需要直接基于当前正式页面判断“放这里还是放那里”。

禁止事项：

- 除应用入口的 `__design` 开发态分流外，禁止为了 `mirror` prototype 修改、抽取、包裹、重组任何正式运行时代码。
- 不得改写正式实现以适配 prototype。
- 不得在 prototype 中重写一份近似正式页面来冒充镜像探索。

实现约束：

- 必须直接原样引用正式页面或正式组件。
- mock 壳层、交互脚本、覆盖层与测试夹具必须全部留在 `design/prototypes/` 或 `__design` 开发态入口，不得进入正式运行路径。
- 若发现“原样引用正式代码”无法完成当前原型目标，先停下收口方案，再切换 `sandbox` 或回 brief，不得先改正式代码去配合 prototype。
- 若镜像原型已经足以说明设计结论，但用户尚未明确要求“落正式实现”，仍不得顺手把结论提前写入正式代码。

#### 沙箱探索模式 `sandbox`

定义：

- 在 `design/prototypes/` 内独立实现一份设计原型，不要求复用正式页面，也不要求贴合当前正式结构。
- 目标是为新页面、大改版、信息架构重组、动效探索或视觉方向探索提供独立实验场。

适用场景：

- 正式页面尚不存在。
- 需要重写整页或大幅调整页面结构。
- 需要同时探索多个视觉方向、布局方向或动效方向。
- 当前正式实现会强烈限制设计判断，继续镜像只会被现状绑死。

禁止事项：

- 不得把 `sandbox` prototype 伪装成“正式页面的现状镜像”。
- 不得把沙箱里尚未拍板的实现直接当作正式代码参考而跳过 brief / 审查。

实现约束：

- 必须明确标注自己是 `sandbox`，不能让评审误以为它就是当前正式页面。
- 不得修改正式代码去配合 `sandbox` prototype。
- 若 `sandbox` 的结论需要回到现有正式页面验证真实布局压力、真实状态承载或真实手势路径，应再补一个 `mirror` prototype 或进入正式实现前的对齐验证。

#### 模式切换规则

- `mirror -> sandbox`：当正式页面结构不足以回答当前设计问题，或出现“为了原型不得不改正式代码”的迹象时切换。
- `sandbox -> mirror`：当大方向已收口，需要验证它落回当前正式页面后是否成立时切换。
- 两种模式可以在同一 feature 下并存，但必须在 brief 中明确各自目标，不得混用口径。

### 阶段门与授权门

这里固定区分两条互不替代的判断轴：

1. 原型方法轴：`mirror / sandbox / none`
   - 只回答“设计探索怎么做”。
2. 交付阶段轴：`design / implementation`
   - 只回答“现在是否允许改正式代码”。

硬规则：

1. 只要还在 `design` 阶段，就算使用 `mirror`、就算方案已经非常明确、就算改动极小，也默认不改正式运行时代码。
2. 进入 `implementation` 阶段必须满足两个条件：
   - 设计结论已经被用户拍板，或用户明确接受以当前 prototype / brief 作为实现依据。
   - 用户明确发出“开始实现 / 落正式代码 / 进入正式实现”这一类授权，而不是仅要求“做原型 / 做镜像 / 给我看设计”。
3. `mirror` 不是“边看边改正式代码”的快捷通道；它只是更高保真的观察方式。
4. 如果 agent 判断“这个改动已经很确定，直接顺手落代码更省事”，也必须先停下，向用户说明将从设计阶段切换到实现阶段，再继续。
5. 若用户只要求优化工作流、审查原型或分析方案，不得借机推进正式实现。

### 阶段 4：可视化审查与拍板

1. 启动开发服务器，给用户提供本次原型的直达 URL，例如 `http://localhost:{port}/__design/{feature}`。
2. 按 brief 对应入口审查 prototype。
3. 用户拍板后，在 brief 中记录最终采用方案与放弃方案。
4. 通过审查的 brief 从 `briefs/active/` 移入 `briefs/accepted/`。
5. 用户提出修改/微调，在当前活跃brief与prototype上重新执行阶段2 3 4。
6. 用户要求推倒重做，放弃方案移入 `briefs/rejected/`。
7. 禁止存在未分类 brief。

### 阶段 5：固化设计基线

**只有在可视化审查通过后，才把稳定结论写入设计基线：**

- 新增或改变组件规则：更新 `design/components/{component}.md`
- 新增或改变用户流程：更新 `design/flows/{flow}.md`
- 新增或改变全局视觉规则：更新 `design/brand/*`
- 新增或改变跨端 / 手势 / 技术约束：更新 `design/standards/*`
- 存在重要取舍：新增 `design/decisions/DDR-xxxx-{title}.md`

### 阶段 6：正式实现

1. 编码实现只能基于以下已拍板设计资产：
   - accepted brief
   - prototype
   - brand
   - components
   - flows
   - standards
   - DDR
2. 若实现过程中发现 design 无法落地，不能直接在代码里变通。
3. 必须先回到 `design/` 修改 brief / prototype / component / flow / DDR，再继续实现。
4. 若本轮此前仍处于 `mirror / sandbox` 设计探索，进入本阶段前必须先获得一次明确的实现授权，不能把“原型已看过”自动视为“同意开始编码”。

### 阶段 7：一致性检查

1. 实现完成后，检查正式 UI 是否与已拍板 design 对齐。
2. 不要求每次都机械反写 design。
3. 但任何实际偏离都必须通过新的 brief / DDR / component / flow 修改显式确认。

## 使用规则

- 每一次 UI/UX 改动必须先从 `design/briefs/active` 开始。
- 没有 accepted brief 或明确 design baseline 引用的 UI 改动，不应直接进入正式代码。
- 只有在可视化审查通过或用户拍板微型修补后，才能把稳定结论写入基线文档。
- 正式实现只能参考已拍板的 design 资产。
- 代码是实现事实，`/design` 是设计源头；两者通过“设计先行 + 实现对齐检查”保持一致。
- 处于设计工作流时，默认优先产出设计资产而不是实现代码；除非用户明确切换到实现阶段，否则不得把设计结论直接落进正式代码。

## 能力边界

- `mirror` 不是用来做大改版的。它最适合已有正式实现上的高保真局部探索。
- `sandbox` 不是用来验证“当前正式页面里这块到底会不会挤压”的。它最适合探索方向，不适合替代落地前的真实页面验证。
- 当问题核心是“现有正式页面上，这个新状态放进去是否成立”，优先 `mirror`。
- 当问题核心是“我怀疑这整页都要换结构 / 换视觉 / 换交互模型”，优先 `sandbox`。
- 当一个问题同时涉及“方向探索”与“现状验证”，应先 `sandbox` 收口方向，再补 `mirror` 验证落回正式页面后的真实表现。

## 开发态预览

- `__design` 仅开发态启用。
- `__design` 根路径只作为 prototype 索引。
- 每个 prototype 必须有独立路径，格式为 `/__design/{feature}`。
- 审查链接必须直达本次需要检查的 prototype，不允许让用户先浏览无关 prototype。
- `mirror` prototype 可直接原样引用正式页面或正式组件；`sandbox` prototype 则完全留在 `design/prototypes/` 内独立实现。
- 两种模式的 mock 数据、状态壳层和审查页面都必须只在 `__design` 开发态路径加载。
- `__design` 不进入正式移动端产物。
- prototype 只服务设计拍板，不作为生产组件。

## 维护约束

- 原 `DESIGN.md` 不再作为维护入口；历史通过 Git 追溯。
- 若主仓库代码现状与历史设计资产冲突，以主仓库代码现状为唯一基准，重新沉淀到本目录。
- 核心文档只保留本工作流的简短入口说明，不复制这里的完整内容。
