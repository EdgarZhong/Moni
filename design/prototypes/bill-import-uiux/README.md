# Bill Import UI/UX Prototype

## 对应 Brief

- `design/briefs/active/bill-import-uiux.md`

## 解决的设计问题

- 验证“现有两个按钮入口不变”的前提下，按钮后的导入状态是否足够清晰。
- 验证微信 / 支付宝只区分 `expectedSource` 与密码文案时，用户是否仍能理解当前导入上下文。
- 验证密码输入改为二级页后，是否能避免挤压随手记主流程。
- 验证原型工作流是否能直接原样引用正式 `MoniEntry`，而不是改写正式代码或手搓近似页面。

## 展示的状态

- 默认态
- 完整记账页入口态
- 微信 / 支付宝触发密码二级页
- 6 位数字密码输入，使用系统数字键盘能力
- 密码错误
- 密码正确后回到记账页导入卡片
- 导入卡片内导入中反馈
- 导入卡片内成功反馈

## Mock 数据说明

- 直接挂载正式 `MoniEntry` 页面。
- prototype 在开发态覆盖正式页面的导入按钮点击，并进入密码二级页。
- 密码正确后会向正式页面自己的隐藏文件输入注入一个 mock 账单文件，让正式页面原样展示成功提示。
- prototype 会在挂载期间临时把 `appFacade.importRawData` 替换为 no-op，避免污染真实账本数据。
- 原型密码固定为 `123456`，用于验证成功 / 错误路径。
- 桌面 F12 移动模拟下可直接用电脑键盘输入密码；真机应弹出系统数字键盘。

## 不代表最终实现的部分

- 具体动效、停留时长与真实错误文案仍需正式实现时按后端返回值对齐。
- prototype 的覆盖层与 mock 注入只在 `__design` 开发态路径存在，不进入正式运行路径。
