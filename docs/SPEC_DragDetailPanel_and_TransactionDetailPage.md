# 规格文档：拖拽细则面板 & 交易详情页

> 版本：v1.1  
> 适用范围：`DragDetailPanel` 组件 + `TransactionDetailPage` 页面  
> 本文档覆盖：状态流转、逻辑设计、展示与交互规格  
> 不覆盖：像素级视觉尺寸、容器排布细节

**边界说明**：本文档只用于补充新增的 `DragDetailPanel`，以及细化 `TransactionDetailPage` 的具体定义；不改写 `docs/Moni_Homepage_Integration_Spec.md`、`docs/Moni_Requirements_v3.md` 等既有固定规格。首页拖拽投放后的可选理由输入仍沿用既有 `ReasonDialog` 口径。

## 一、字段优先级参考

本文档所有展示逻辑均基于以下字段语义，以账本 JSON 记录为准：

| 字段 | 语义 | 备注 |
|---|---|---|
| `counterparty` | 交易对方名称（商户/人名）| 最强身份标识 |
| `product` | 交易描述 / 备注 | 常含冗余前缀如"收款方备注:" |
| `rawClass` | 支付平台原始分类 | 如"扫码支付"、"转账"、"二维码收款" |
| `time` | 交易时间 | `YYYY-MM-DD HH:mm:ss` |
| `sourceType` | 账单来源 | `wechat` / `alipay` / `manual` |
| `amount` | 金额（数值） | 单位：元 |
| `direction` | 收支方向 | `in`（收入）/ `out`（支出）|
| `paymentMethod` | 支付方式 | 零钱、银行卡等 |
| `transactionStatus` | 交易状态 | 通常为 `SUCCESS` |
| `remark` | 备注字段 | 导入账单时可来自原始账单；当前系统也复用该字段承接详情页“备注”编辑 |
| `ai_category` | AI 给出的分类 | 可为空 |
| `ai_reasoning` | AI 的推理理由 | 可为空 |
| `user_category` | 用户确认的分类 | 可为空 |
| `user_note` | 用户填写的修正理由 | 可为空 |
| `is_verified` | 是否已锁定 | `boolean` |
| `updated_at` | 最后更新时间 | |

**展示标题（display title）计算规则**（适用于本文档所有场景）：

```
if counterparty is not empty:
  displayTitle = counterparty
elif product is not empty:
  displayTitle = stripRedundantPrefix(product)   // 剥离"收款方备注:"等前缀
else:
  displayTitle = rawClass ?? "未知交易"
```

`stripRedundantPrefix` 规则：若 product 以 "收款方备注:"、"商品:"、"备注:" 等常见前缀开头，去除该前缀后作为展示文本。

## 二、拖拽细则面板（DragDetailPanel）

### 2.1 概念模型

面板在拖拽蒙版激活期间始终存在于屏幕底部，共有三种状态：

- **Hidden**：当前不在拖拽模式，面板不渲染
- **Collapsed（收缩条）**：拖拽激活，面板以薄条形态固定在屏幕底部，仅展示最精简的信息，不遮挡分类格
- **Expanded（展开）**：手指进入感应区，面板向上展开，同时将上方分类区整体向上顶出屏幕

面板始终是**纯展示组件**，不承载任何可交互操作。拖拽投放后的可选理由输入继续沿用既有 `ReasonDialog`；其余完整编辑入口在交易详情页。

**底边锚定原则**：无论收缩态还是展开态，面板的底边都应持续锚定在屏幕下边缘之外，视觉上表现为“从屏幕底下长出来”，而不是一张完整卡片整体上移到屏幕内。展开时只允许面板向上增高，不允许面板整体脱离底边。

**三区分层原则**：整个拖拽蒙版按纵向固定拆成三个连续区域，禁止再引入额外的“安全区外层”“补偿位移层”等旁支结构。

1. **上半区：分类区**
   - 承载全部分类卡片
   - 容器最大高度保持当前基线，不因展开态被重新拉高
   - 区内单个分类卡片改为自适应高度，只根据自身文案和内边距决定高度
   - 分类卡片布局继续保持明确的两列栅格；即便总数为奇数，最后一项也只占单个半宽格子，不允许独占整行
   - 不要求分类卡片把上半区完全填满；若未来分类更多，允许在该固定最大高度内继续换行或滚动
2. **中间窄安全带**
   - 高度很窄，只承担“分类区”和“细则区”的中性缓冲
   - 不做任何视觉样式，不加描边、底色、渐变、阴影或提示元素，保持为空白带
   - 手指停留在这里时，不命中分类卡，也不触发细则展开
   - 该区域存在的目的，是给用户一个可暂停、可取消、可重新判断方向的落点
3. **下半区：底部详情面板**
   - 底边始终锚定在屏幕底部外侧
   - Collapsed / Expanded 的视觉变化只发生在这个区域内
   - 驻留区、细则区、展开阈值都归这个区域负责

**展开态的空间分配原则**：展开态的详细信息内容集中在面板的中部至中上部区域，面板底部预留一段空白区域，以虚线框标识为"拖拽驻留区"。这一设计有三个目的：
1. 手指停留在屏幕下方时不遮挡信息内容
2. 虚线框作为视觉语言，直觉传达"手指在这里 → 面板展开；手指离开这里 → 面板收起，可继续分类"
3. 缩短用户在查看详情与返回分类操作之间所需的手指移动距离

---

### 2.2 状态流转

```
拖拽激活 (longpress confirmed)
  └─→ 面板进入 Collapsed 态，从屏幕底部 slide-in

拖拽进行中（持续监听 pointermove 的 clientY）：

  clientY ≥ detailPanelTopY
    └─→ 切换到 Expanded 态
        面板保持底边锚定，通过增高让顶边上移 expandDelta
        分类区与中间安全带同步向上 translate(expandDelta)

  clientY < detailPanelTopY
    └─→ 切换到 Collapsed 态（若当前为 Expanded）
        面板、分类区与中间安全带同步还原

拖拽结束（松手投放 or 取消）
  └─→ 面板 slide-out 退出，回到 Hidden 态
```

**展开阈值定义**：只有当手指 y 坐标真正进入细则区顶边（`detailPanelTopY`）以下时，才触发展开；停留在分类区与细则区之间的窄安全带时，不触发任何形态切换。

**对齐校正说明**：若实现中出现“用户手指仍停留在希望查看细则的位置，但视觉驻留区看起来还在更下方”的错位感，应优先下调触发展开阈值或上提驻留区内部内容，而不是把驻留区整体继续压向面板底部。

**关键规则——安全带必须保持中性**：判断基准始终是细则区顶边 `detailPanelTopY`，而不是提前向上外扩到安全带里。这样用户可以把手指停在安全带中重新判断，而不会被提前拉进 Expanded。

**`detailPanelTopY` 的确定**：该分界线是父层里的固定几何值，不通过子组件 DOM 实时测量回推。实现上应直接按当前视口高度减去收缩态面板可见高度计算：

```typescript
detailPanelTopY = window.innerHeight - collapsedPanelVisibleHeight
```

其中 `collapsedPanelVisibleHeight` 即 Collapsed 态在屏幕内真实可见的那段高度。此后整个拖拽过程中判定阈值始终以该固定分界线为基准；展开时即便分类区与细则区整体发生位移动画，也不改变这条判定线。

---

### 2.3 动画规格

**Collapsed 进场**（拖拽激活时）：
- 面板从屏幕底部外侧 translate(100%) → translate(0)
- Duration: 220ms，easing: ease-out

**Collapsed ↔ Expanded 切换**：
- 面板和分类区**共享同一个纵向位移量**，delta 相同，时序相同；但实现上必须保持为两个独立区域，各自只处理自己的位移与内容布局，不允许为了“彻底退出屏幕”再给分类区额外追加第二套位移
- 三段区域在静止态与动画态都必须保持**紧密拼接且互不覆盖**，中间不再额外插入补偿空隙
- Duration: 240ms，easing: ease-out；不使用弹簧或回弹曲线，要求平滑到位，不制造额外的物理弹跳感
- 面板与分类格之间的间距在整个动画过程中保持固定不变

**Expanded 展开量（expandDelta）**：
- `expandDelta = expandedHeight - collapsedHeight`
- 面板自身不整体上移，而是保持底边锚定，通过增高让顶边上移 `expandDelta`
- 分类区向上移动 `expandDelta`
- 分类区允许在展开态被明显顶出屏幕；但其自身最大高度保持稳定，不借由展开态额外膨胀
- 视觉效果上必须呈现“从屏幕底部向上长出”；但具体实现可优先采用 `transform` / 合成层动画来模拟这一效果，避免依赖大面积 `height` 变化导致拖拽时卡顿

**退场**（拖拽结束）：
- 面板 translate(0) → translate(100%)，同时分类格还原
- Duration: 180ms，easing: ease-in

**性能约束**：
- 拖拽激活、Collapsed ↔ Expanded 切换、拖拽结束三个阶段都不应出现明显掉帧
- 优先使用合成层友好的动画属性（如 `transform`、`opacity`），谨慎对大面积容器使用 `height`、`top`、`bottom`、`box-shadow` 的连续动画
- 若视觉精修与动画性能冲突，以拖拽流畅性优先
- 长按刚成立时不得出现与真实手指位置无关的“先向外弹一下”现象；进入 Collapsed 的进场动画与 Expanded 判定必须解耦，避免面板因为初始测量、动画冲突或首帧误判而瞬间外弹

---

### 2.4 Collapsed 态展示规格

收缩条空间极为有限，仅展示让用户快速识别"这是哪笔钱"以及"当前分类是什么"所需的最少信息。

**左侧**：
- `displayTitle`，单行，超出后 `…` 截断（约 12–14 个中文字符宽度）

**右侧**：
- `direction` + `amount`，格式化为"支出 ¥X.XX"或"收入 ¥X.XX"

**标题下方（副行）——当前分类结果**，按如下优先级取值：

```
if user_category 非空:
  展示 user_category 标签
  if user_note 非空:
    标签右侧或下方展示 user_note（单行，超出截断）
elif ai_category 非空:
  展示 ai_category 标签（视觉上区别于 user_category，表示"AI 建议，未确认"）
  if ai_reasoning 非空:
    标签右侧或下方展示 ai_reasoning（单行，超出截断）
else:
  展示"未分类"警示标签
```

**不展示**：is_verified 状态、time、product、rawClass、paymentMethod 等。Collapsed 态信息密度不超过两行，不引入任何图标以外的装饰元素。

**补充提示**：
- Collapsed 态底部允许增加一行弱提示文案，例如“拖到此处查看交易细则”，帮助用户理解下半区的用途

---

### 2.5 Expanded 态展示规格

展开后面板占据屏幕较大纵向空间。内容布局遵循以下原则：**所有信息内容集中在面板中部至中上部，面板底部留白并以虚线框标注拖拽驻留区**。

**底部虚线框（拖拽驻留区）规格**：
- 固定高度，位于展开态面板的最底部
- 以明显的虚线边框矩形（圆角）直接框出整个“手指停留区”
- 虚线框顶部应尽量贴近真实展开阈值线；若视觉上仍然低于手指实际停留区，应优先下调 `EXPAND_TRIGGER_MARGIN`，并把虚线框整体上提，而不是继续把判定区藏在屏幕外
- 可配合极简提示文案或箭头图标，直接传达“停留看细则 / 上移去归类”的语义
- 此区域不响应任何点击或手势事件
- 虚线框的可见位置应与真实的展开/收起判定区尽量对齐，避免出现“视觉承接区”和“实际判定区”上下错位
- 驻留区高度应足以在视觉上承接拖拽预览卡片；展开态下，预览卡片应明显落在该区域内部，而不是漂浮到信息内容区中央
- 不应在驻留区最底部额外再画一个更小、位置更低的“伪停留框”来替代整块区域，否则用户会误判真实的收起边界
- 面板本体仍需保持与屏幕底边视觉连续；实现上允许面板底部继续延伸到屏幕外，但**虚线驻留区本身应完整落在可视区内**，不要让驻留区底边继续沉到屏幕外
- 细则区与驻留区之间不应出现大块无效白场；若当前交易细则较少，可通过收紧展开高度、提升驻留区位置、增加过渡色带等方式缓解，但不要牺牲驻留区的承接清晰度
- 驻留区本身允许使用轻量色彩或渐变增强识别，但色彩必须服务于“承接拖拽预览卡片”的语义，不能喧宾夺主
- 驻留区整块背景、描边与提示色统一收口为主题青色系，不再混入红色系强调色

**信息内容区**（位于虚线框上方，展开态面板中上部）：

展开态的目标不是展示分类结果，而是要在当前首页读模型可提供的范围内，**尽量完整展示这笔交易本身的细则**，帮助用户回忆并判断它应该被拖到哪个分类。

**块 A：交易身份与金额**

- `displayTitle`：大号主标题，允许换行；计算顺序沿用本规格前文定义，即优先 `counterparty`，其次为剥离冗余前缀后的 `product`，再其次为 `rawClass`
- `direction + amount`：大号金额，格式同 Collapsed 态
- `sourceLabel / 完整时间 / sourceType`：以紧凑标签或次级元信息形式展示，其中时间必须写成“X月X日 HH:mm”，而不是只保留“HH:mm”

来源标签固定口径：

1. 文案统一为：`微信` / `支付宝` / `随手记`
2. 颜色统一为：微信绿、支付宝蓝、随手记黄
3. 该来源标签语义与首页日卡保持完全一致，不得在详情页或拖拽细则里另起一套文案

**块 B：交易细则**

- `counterparty`：若与主标题不同，优先展示，帮助用户回忆交易对象
- `product`：若与主标题存在差异，应单独展示
- `rawClass`：原始账单分类字符串，通常对判断“这笔钱是什么”很有帮助
- `paymentMethod`
- `transactionStatus`：若不为成功状态，应明确展示
- `remark`：若存在，展示备注；若为空可不展示或显示“暂无备注”
- 其他当前首页读模型已具备、且能帮助识别交易来源与属性的字段，也应优先放在这里

**字段优先级建议**：

1. 交易时间（完整时间）
2. 交易对方
3. 商品名称 / 商品说明
4. 原始分类
5. 支付方式
6. 交易状态（仅异常态）
7. 备注

不应把 `is_verified`、`currentCategory`、`aiCategory`、`ai_reasoning`、`user_note` 这类“分类结果或确认状态”信息塞回 Expanded 态顶部，避免再次挤占交易细则空间。

**块 C：分类信息原则**

- 展开态不再展示 `currentCategory / aiCategory / categoryStatus / ai_reasoning / user_note`
- 这些内容属于“当前分类结果”的范畴，已经在 Collapsed 态承担，不应挤占 Expanded 态的交易细则空间

**内容完整性原则**：

- 在当前首页读模型字段允许的范围内，Expanded 态应优先“交易细则完整”而不是“分类结果完整”
- 若未来首页读模型补入更多原始字段（如 `product`、`rawClass`、`transactionStatus` 等），可继续追加到信息内容区，但不应删除上述已确认字段

### 2.6 与拖拽系统的接口约定

面板消费的数据来源是 `draggingTransaction: Transaction | null`，由 DragOverlay 状态管理层提供。

```typescript
interface DragDetailPanelProps {
  transaction: Transaction | null     // null 时面板为 Hidden 态
  panelState: 'collapsed' | 'expanded'  // 由父层根据 drag y 坐标计算后传入
}
```

**拖拽预览卡片补充约束**：
- 预览卡片默认继承首页原条目的图标语义，不应在拖起后统一退化成 `?`
- 若当前条目已有分类，则继续使用该分类对应的图标、底色与强调色
- 仅当原条目本身就是未分类态时，才允许继续显示问号占位

面板自身不监听 pointer 事件（`pointer-events: none`）。expand/collapse 的触发判断由管理拖拽状态的父组件在全局 `pointermove` 中统一完成：

```typescript
const detailPanelTopY = window.innerHeight - collapsedPanelVisibleHeight
const expandArmDistance = 12  // px

// 在全局 pointermove 回调中调用
function updatePanelState(clientY: number, activationY: number) {
  const armed = clientY - activationY >= expandArmDistance
  if (armed && clientY >= detailPanelTopY) {
    setPanelState('expanded')
  } else {
    setPanelState('collapsed')
  }
}
```

**注意**：`detailPanelTopY` 必须和父层里“安全带 / 细则区”的固定分界线一致。它不跟随 Expanded 后分类区整体上推而变化，也不允许通过动画中的 DOM `getBoundingClientRect()` 结果反推。

## 三、交易详情页（TransactionDetailPage）

### 3.1 概念模型

详情页是对单条交易的全功能操作面板，从右侧滑入覆盖首页。它同时承担两个职责：

1. **信息查阅**：展示来自原始账单的所有有用字段
2. **分类精修**：修改分类、填写修正理由、切换锁定状态、添加备注

详情页是用户对"这笔钱"进行完整查阅与编辑的主入口；它不替代拖拽投放后的既有可选理由输入。

---

### 3.2 进入与退出

**进入**：
- 首页流水列表单击条目 → 详情页从右侧滑入（`translateX(100%) → 0`）
- 首页流水列表在详情页打开期间变暗（半透明遮罩），但不完全不可见
- 详情页打开后接管整块屏幕；首页 Root 的 header / footer 不再占据它的可视高度

**退出**：
- 点击左上角返回按钮
- 当前不要求侧边滑动返回。Android 侧边返回手势依赖 Capacitor / 原生集成后再议；在当前系统与原型阶段，统一使用左上角返回按钮
- 退出动画：`translateX(0) → translateX(100%)`

**数据写入时机**：采用**即时写入（auto-save）**策略：
- 每个可编辑字段在变更发生时立即调用 `LedgerService` 写入
- 不设独立"保存"按钮
- 退出不询问确认，所有变更在退出前已持久化

### 3.2.1 画布与 Chrome 归属

- 详情页属于“二级页面中的全屏覆盖页”，视觉上**无全局 header、无底部导航**。
- 详情页顶部只保留它自己的返回标题区；该标题区负责消费统一的顶部基础留白与 `env(safe-area-inset-top)`。
- 详情页的内容滚动区、底部安全带、退场动画都由详情页自身负责，不共享首页 Root 的可视布局高度。
- 为了切页时保留账本状态、AI 状态或返回栈，相关状态宿主可以常驻在更高层；但这不改变详情页“整页接管画布”的视觉语义。

### 3.3 页面区域结构（当前已确认排版顺序）

详情页当前阶段已收口为以下**从上到下的固定阅读顺序**：

1. **Header 区**
2. **交易原始信息区**
3. **AI 分析区**
4. **分类操作区（含锁定控制）**
5. **理由补充区**
6. **系统元数据区**

**排版原则补充**：

- 原始交易信息必须放在最上面，保证用户一进页先看清“这笔钱本身是什么”
- AI 分析区紧跟原始交易信息之后，帮助用户在人工判断前先快速理解系统当前看法
- 分类操作区位于页面中段核心位置；锁定控制不再拆成独立大区，而是并入分类操作区
- 理由补充区只承接“为什么这样分”，不再承接第二个独立的可编辑备注输入
- 系统元数据统一沉到页面最下方，不再混入顶部主信息阅读流
- 整体布局必须明显提高信息密度，优先争取“首屏完成主要查阅与操作”；除极长标题、极长 AI 理由或用户主动展开输入外，默认不应让用户一进页就需要连续下滑很长距离
- 视觉语言必须更像正式面向用户的产品页，而不是开发态信息面板；分类、AI 状态、来源等关键信息应配合图标或图形符号，而不是几乎纯文字堆叠
- 区块标题前的识别符必须使用真正的 icon / 图形符号，不接受用“原 / AI / 分 / 注”这类单字占位充当图标

---

### 3.4 Header 区

- 返回按钮（左），样式与设置页子页面、记账页导入密码页的返回按钮语言统一
- 标题文字："交易详情"（固定文案）
- Header 当前不再重复展示 `dayLabel / sourceType` 等副标题信息；这些信息已经在下方原始交易信息区中承接，顶部应尽量简洁
- 右侧：只保留轻量状态展示，不再承担主要锁定操作；锁定主操作统一放在分类操作区内的开关控件
- 右侧锁定状态展示必须使用锁 / 解锁图标，而不是单字文案；状态切换时允许配一段轻量过渡动画，帮助用户感知“已锁定 / 已解锁”变化

### 3.5 交易原始信息区

此区域为纯展示，不可编辑。展示来自账单原始数据的所有有用字段。

**子区 A：核心身份**

| 字段 | 展示文案 | 条件 |
|---|---|---|
| `displayTitle` | 大号主标题，允许换行，不截断 | 始终 |
| `product`（剥离前缀后）| 次级描述，若与 displayTitle 相同则不重复展示；字号应略高于正文信息卡，但仍低于主标题 | 不同时展示 |
| `rawClass` | "交易类型：XX" | 始终 |
| `time` | "交易时间：YYYY年M月D日 HH:mm" | 始终 |
| `sourceType` | 图标 + 文字（微信 / 支付宝 / 随手记）| 始终 |

**子区 B：金额明细**

| 字段 | 展示文案 | 条件 |
|---|---|---|
| `direction` + `amount` | "支出 ¥X.XX" 或 "收入 ¥X.XX"，大号数字突出 | 始终 |
| `paymentMethod` | "支付方式：XX" | 始终 |
| `transactionStatus` | "交易状态：XX" | 仅当 ≠ "SUCCESS"；SUCCESS 不展示（正常态，无信息量）|
| `remark` | "备注：XX" | 仅当非空且非 "/" |

**边界收口**：

- 此区只展示“用户理解这笔交易”所必需的原始信息
- 对于当前读模型中所有**非空的原始细则字段**，详情页原始信息区必须做到“信息不遗漏”：
  - 可以因为已被 `displayTitle`、次级描述或顶部信息 badge 承接而不重复显示
  - 但不能因为排版偷懒而直接省略
- 标题、次级描述、顶部 badge、信息卡四层之间必须做去重：
  - 已用于 `displayTitle` 的字段，不再在下方重复
  - 已用于次级描述的字段，不再在下方重复
  - 已用于顶部来源 / 时间 / 状态 badge 的字段，不再在信息卡重复
  - 同值字段（如 `sourceLabel` 与 `sourceType` 用户文案一致）只保留一处
- 状态信息视为原始细则的一部分：
  - `transactionStatus` 只要非空就必须展示
  - `SUCCESS` 不再视为空；但允许在顶部 badge 里以低强调样式展示，而不是强制落成大卡片
- 顶部来源 / 时间 / 状态等 badge 必须固定在同一行内，不允许自动换行
  - 当可用宽度不足时，badge 行允许横向滚动查看
  - 不得因为响应式压缩而把 badge 挤成两行
- 原始细则卡片区采用 **6 栏自适应网格**，而不是简单两列：
  - 允许的卡片跨度仅为 `2 / 3 / 4 / 6`
  - 典型短字段如 `rawClass` 优先占 `2`
  - 典型中字段如 `paymentMethod`、较短 `counterparty / product / billTitle` 优先占 `3` 或 `4`
  - 长文本、换行文本、低优先级但信息量大的补充字段（如长 `product`、长 `billTitle`、`remark`）占 `6`
- 每一行都必须被占满，不允许留下视觉残缺的空槽：
  - 合法组合只允许 `2 + 4`、`3 + 3`、`2 + 2 + 2`、`6`
  - 优先通过调整相邻卡片跨度来补满整行，而不是简单把最后一张卡强拉成整宽
  - “落在最后一行”本身**不是**整宽条件；只有内容本身足够长、换行较多、或语义上更适合下沉到底部时，才允许占满整行
- 当去重后只剩下一个**极短字段**时：
  - 只有“支付方式 / 交易状态”等纯辅助状态字段允许降级为轻量 badge / capsule 展示
  - `rawClass`（原始分类）不允许被提到顶部 badge 行，必须继续留在原始信息卡片区
  - 禁止把这类短字段单独拉成整宽信息卡
- 建议的剩余细则展示顺序固定为：
  - 第一层：`displayTitle`
  - 第二层：次级描述（例如未被标题消费的 `product`）
  - 第三层：顶部 badge（来源 / 时间 / 状态 / 极短单例字段）
  - 第四层：原始细则网格，字段顺序优先为 `paymentMethod -> rawClass -> counterparty -> product -> billTitle -> remark`
- 所有标题、次级描述、badge 文案与信息卡内容一律允许换行，不得因固定高度或 `nowrap` 导致文字冲出卡片
- `updated_at / id / originalId` 等系统元数据不再放在这里，而是统一下沉到页面最底部的“系统元数据区”

### 3.6 AI 分析区

此区域为纯展示，不可编辑。

**展示位置**：固定放在“交易原始信息区”之后、“分类操作区”之前。

**展示条件**：`ai_category` 非空时展示此区域；若为空则展示占位文字"AI 尚未分析这笔交易"。

**展示内容**：

| 字段 | 展示文案 |
|---|---|
| `ai_category` | 仅展示分类标签本体，不再重复写 "AI 分类：" 前缀 |
| `ai_reasoning` | 完整展示，不截断，允许多行 |

**若 user_category 与 ai_category 不同**：在此区域顶部加一行提示："你已将分类修改为 [user_category]"，视觉上区分"AI 的原始判断"与"当前生效分类"。

### 3.7 分类操作区（含锁定控制）

此区域是详情页最核心的操作区。

**当前排版收口**：

- “当前分类 / 用户分类入口”与“锁定开关”必须位于同一个操作区内
- 锁定开关应与分类入口形成同级、邻近关系，而不是沉到页面底部成为独立控制区
- 在移动端默认详情页中，不再拆成“左侧分类卡 + 右侧锁定卡”两块
- 应改为一张整宽分类卡：卡片顶部一行左侧为“当前分类”标题，右侧为“锁定此分类”与开关，开关固定贴右
- 分类入口与当前分类结果都必须带有明确图标语义，不接受纯文字标签孤立悬空
- 锁定说明文案不再塞进右侧紧凑锁定卡内部；应下沉到并排布局下方，作为分类操作区底部的整宽说明
- 锁定说明文案必须继续保留，直接说明锁定的实际作用
- 该区整体要明显比上一版更紧凑，不允许再出现大块留白和整屏分类按钮墙
- 若页面顶部另有锁定状态标识，其图形语言必须与本区锁定语义一致，不得出现一处是图标、一处是纯文字的割裂表达

**当前分类展示**：

- 若 `user_category` 非空：展示已确认分类标签 + 分类图标，辅助来源文案只保留"来自用户"
- 若 `user_category` 为空但 `ai_category` 非空：展示 AI 分类标签（区别样式），辅助来源文案只保留"来自 AI"
- 若两者均为空：展示"未分类"警示态标签，辅助文案维持极简，不再堆叠解释性句子

**修改触发**：
- 点击当前分类标签区域（任何状态均可点击）→ 弹出紧凑分类选择器（优先使用图标化、面向用户的弹层选择器，不再默认平铺整页分类按钮）
- 选择新分类后，选择器关闭，`user_category` 立即更新
- 分类更新后，页面应自动平滑滚动到下方 `user_note` 输入区，主动露出"告诉 AI 为什么"区域，提示用户补充原因
- 若新分类 ≠ `ai_category`（或 ai_category 为空）→ 自动生成 B 类学习实例（AI 建议被覆盖）
- 若新分类 = `ai_category` → 自动生成 A 类学习实例（用户确认了 AI 的建议）
- 分类变更后，`is_verified` 自动设为 `true`（用户主动操作代表确认意图）
- 分类选择器弹层的最外层卡片必须真正裁切圆角，不能出现头部 / 内容区 / 底部背景从圆角外侧露出直角尖角
- 分类选择器弹层的字体必须跟随 `font-brand`，不得回退到系统衬线或其它未在设计体系中的字体族

**交互冻结补充**：

- 详情页内不再直接平铺整组分类按钮作为默认主界面
- 默认界面只展示“当前分类结果 + 点击修改入口”；真正选择分类时再弹出选择器

**分类变更的副作用链**：

```
user_category 变更
  → 写入 LedgerService
  → 生成 A 或 B 类实例，写入实例库
  → is_verified 自动置 true（若分类变更）
  → 若实例库新增数量达到学习阈值，触发学习会话
```

---

**锁定控制并入规则**：

- 锁定控件属于分类操作区的一部分，不再单独拆成第六大区
- 锁定控件采用接近 iOS 设置页的横向开关样式，而不是图标按钮或文字块按钮
- 锁定标签文案继续使用"锁定此分类"
- 锁定控件下方说明文字当前固定为："锁定后，系统不会再自动改写这条记录的分类；除非你主动修改或手动解锁"
- 该说明必须面向普通用户直接表达，不使用“冻结边界”“自动化写回保护”等内部术语

**状态逻辑**：

| 触发方式 | is_verified 变化 | 备注 |
|---|---|---|
| 用户在详情页手动分类 | 自动置 true | 用户主动确认 |
| 用户在拖拽界面完成投放 | 自动置 true | 同上 |
| 用户手动关闭锁定按钮 | 置 false | 解除冻结，条目重新回到可自动处理状态 |
| 用户手动开启锁定按钮 | 置 true | 明确锁定当前分类 |
| AI 批量分类写回 | 不修改 is_verified | AI 只写 ai_category，不碰锁定状态 |

**写入时机**：锁定按钮切换时立即写入 `LedgerService`。

**补充口径**：

- 锁定保护独立于“用户输入优先于 AI”的仲裁优先链；即使条目当前没有 `user_category`，只要被锁定，也不能接受自动化分类结果写回。
- 锁定条目默认不参与自动重分类任务的生产判定，但若其所在日期因为其他未锁定条目而入队，分类会话仍可看到这条交易的完整上下文。
- 自动解锁只允许发生在以下显式路径：
  - 用户手动关闭锁定按钮
  - 标签删除导致原分类语义失效时，对应条目强制解锁
  - 修改标签描述并选择“该标签下所有交易”时，用户在锁定列表中当场勾选解锁
  - 设置页执行“全量重分类”时，用户在锁定列表中当场勾选解锁

### 3.8 理由补充区

此区域当前只保留一个可编辑字段。

**修正理由（user_note）**：

- 区块标题保留"告诉 AI 为什么"
- 区块内部不再重复写第二遍标题，进入该区后直接展示输入框
- 多行文本输入框

- 此字段内容会影响实例库：若 `user_note` 非空，对应实例记录中附加该理由，供学习会话提炼
- 键盘弹出时页面上推，确保输入框不被遮挡
- 默认界面不额外展示“这段内容会进入学习链路”之类的解释性辅助文案，避免把表单区重新做成说明书

## 四、两个功能共享的注意事项

### 4.1 实例库写入时机一致性

无论通过拖拽投放还是详情页修改，分类变更后的实例生成逻辑完全一致：

```
分类来源为 user_category 写入，且与 ai_category 对比：
  - 相同 → A 类实例（正向确认）
  - 不同 → B 类实例（纠错，附带 user_note 如有）
  - ai_category 为空，user_category 非空 → C 类实例（补全）
```

D 类实例（随手记）不在此两个功能的范围内。

### 4.2 指针捕获与手势兼容

DragDetailPanel 在拖拽蒙版激活期间存在。面板本身不捕获 pointer 事件（`pointer-events: none`），所有 pointer 事件仍由父层的全局 `pointermove` / `pointerup` 监听处理，确保不干扰现有拖拽系统。

### 4.3 LedgerService 调用方式

详情页的所有写入均通过 `LedgerService` 完成，禁止详情页组件直接读写 JSON 文件。当前字段写入应对齐现有服务接口：

- `LedgerService.updateCategory(txId, userCategory, userNote?)` → 写分类；若同时传入理由，则一并承接 `user_note`
- `LedgerService.updateUserReasoning(txId, userNote)` → 仅写 `user_note`
- `LedgerService.updateRemark(txId, remark)` → 仅写 `remark`
- `LedgerService.setVerification(txId, isVerified)` → 仅写 `is_verified`

分离的目的是保持实例生成逻辑的触发条件明确可控，避免“备注”写入误触发学习链路。
