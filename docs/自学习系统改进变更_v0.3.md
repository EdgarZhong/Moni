# Moni 第一轮低成本精度优化需求变更说明 v0.3

> 当前核心目标:在现有 Moni 机制上，用可控工程成本提升分类精度；优先让上下文更有效、触发更聪明、模型调用更经济。最终追求：精度尽量高，对模型能力依赖尽量低，token 消耗可控。

## 1. 背景与本轮实践发现的问题

Moni 当前的核心能力是：用户导入账单数据后，系统调用 LLM 对账单条目进行分类，并根据用户后续纠错结果持续学习，从而形成个性化分类能力。

本轮实践中，Moni 暴露出的主要问题不是单一分类错误，而是现有学习机制、上下文注入机制、模型参数配置和记忆生成方式之间存在若干低成本可修复的缺陷。第一轮优化目标不是重构完整 Agent Memory 系统，而是在现有机制上优先修复这些直接影响精度和稳定性的工程问题。

### 1.1 模型输出稳定性不足

如果模型运行温度偏高，同一批账单、同一套上下文下，分类结果和理由表达可能出现不必要的漂移。账单分类属于结构化判断任务，不需要太强的创造性输出，因此应优先降低随机性，提高可重复性和一致性。

### 1.2 推理模型 / thinking 参数是否真正生效需要确认

当前系统使用统一的自实现 LLM client。实践中需要确认：系统是否真正调用了推理模型，thinking / reasoning 参数是否被正确传递，供应商侧是否实际启用了推理模式。

如果实际仍在使用普通 chat 模式，复杂场景下的分类、学习和规则总结能力会受到影响。

### 1.3 学习会话只看增量，容易缺少近期语境

如果学习会话只注入本轮新增纠错，模型容易孤立地理解少量样例，无法看到用户近期的完整分类习惯。这会导致学习结果过度概括，或者把偶发纠错误认为稳定规律。

例如，某类账单在近期多次被用户纠正，但如果学习会话只看到其中一两条，就难以判断这是高频模式还是单次例外。

### 1.4 分类会话缺少稳定的近期实例注入

用户刚刚纠正或确认过的样例，往往对下一轮分类有直接价值。若分类会话只依赖旧记忆或普通检索，近期偏好可能不能及时反映到分类结果中。

因此，分类会话需要稳定注入最近实例。同时，当前系统已有正例 / 反例分块机制，新增最近实例时也应保持这一结构，而不是混成一块。

### 1.5 无 userNote 的纠错容易被模型强行解释

用户纠正分类时，不一定会填写 `userNote`。没有备注的纠错只能说明“用户最终把该条目归为某类”，不能说明“为什么”。

如果模型根据无备注纠错强行总结原因，就可能生成伪规律。例如，某条特殊消费被错误总结成长期偏好，进而污染后续学习和分类判断。

因此，无 `userNote` 的实例需要被明确降级为弱证据，并且这个弱证据标记不能只在学习会话里体现，也要在分类会话注入时体现。

### 1.6 学习会话可能生成依赖未来 userNote 的不可执行规则

学习会话中，历史实例的 `userNote` 是可见的；但未来待分类账单通常没有用户事后备注。

如果学习会话生成类似规则：

```text
如果用户备注包含“和女朋友”，则归为正餐
如果用户注释说明是补差价，则归为某类
```

这类规则在未来分类阶段不可执行，因为待分类条目通常不存在用户纠错备注。

这属于学习阶段可见信息和分类阶段可见信息不一致的问题。学习 Prompt 需要明确约束模型：`userNote` 可以帮助理解历史纠错原因，但不能被当作未来待分类条目的可见字段。

### 1.7 少量关键纠错可能因未达到阈值而没有及时学习

如果学习触发机制依赖固定阈值，例如累计 5 条才学习，那么 1–4 条高价值纠错可能迟迟无法进入学习流程。用户再次开启分类时，系统仍然使用旧记忆，容易重复犯错。

因此，在用户开启分类前，应检查是否存在未学习实例；如果存在，应先强制触发一次学习，再开始分类。

### 1.8 记忆条目容易混写“具体实体经验”和“泛化规律”

当前记忆库尚未结构化区分“实体记忆”和“规律记忆”。实践中观察到，模型可能把具体实体经验和泛化规律写进同一条记忆。

例如：

```text
水之源川菜多次为正餐，因此川菜馆等餐厅……
```

这类记忆同时包含两种不同层级的信息：

```text
实体经验：水之源川菜这个具体商户如何分类
泛化规律：川菜馆这一类商户如何分类
```

混写会带来两个问题：

1. 具体个性化信息被稀释；
2. 单个实体经验被过早泛化成类别规律。

第一轮暂不改记忆库结构，但需要先通过学习 Prompt 约束模型：一条记忆只能表达一种类型；实体经验和泛化规律不得混写。

## 2. 第一轮确认实施项

### 2.1 默认模型运行温度设为 0.2，并保留设置页可调

#### 当前问题

如果模型调用温度偏高，同一上下文下分类结果和理由表达可能漂移，影响分类稳定性。

#### 新要求

系统默认模型运行温度设为：

```text
temperature = 0.2
```

适用范围包括：

```text
分类会话
学习会话
后续检查会话
其他与分类 / 学习相关的核心 LLM 调用
```

同时，该参数应保留为可配置项，允许后续暴露到设置页面中供用户或开发者调整。

#### 预期效果

降低输出随机性，提高分类和学习结果的一致性，同时保留后续调参空间。

---

### 2.2 当前阶段统一使用推理模型，并检查 thinking 参数是否生效

#### 当前问题

目前系统使用统一的自实现 LLM client。是否真正启用了推理模型或 thinking / reasoning mode，需要进一步确认。

#### 新要求

第一阶段暂不做普通模型 / 推理模型分流。核心会话先统一使用推理模型。

需要检查：

```text
LLM client 是否正确传递 thinking / reasoning 参数
供应商 API 是否实际接受该参数
日志中是否能确认 reasoning mode 已启用
如果使用 DeepSeek 系列，需要确认调用的是 reasoner 模式，而不是普通 chat 模式
```

#### 预期效果

先排除“以为使用了推理模型，实际没有开启”的配置问题。

---

### 2.3 学习会话注入“待学习增量 + 最近 30 条实例”

#### 当前问题

学习会话如果只看到本轮增量纠错，容易缺少近期用户分类习惯，导致学习结果过度概括。

#### 新要求

学习会话输入改为：

```text
待学习增量实例
+ 最近 30 条实例
```

最近 30 条实例直接注入，不做轻量摘要。

实例应尽量包含：

```text
交易基础信息
原分类 / AI 分类
用户最终分类
userNote，如果存在
纠错时间或确认时间
金额、时间、对手方、备注等必要字段
```

#### 预期效果

让学习会话看到近期用户真实偏好，减少孤立学习和过度泛化。

---

### 2.4 分类会话注入最近 30 条实例，并保持正例 / 反例分块

#### 当前问题

用户近期纠正或确认过的样例，对下一轮分类有很高价值。若分类会话只依赖旧记忆或普通检索，近期偏好不能及时影响分类。

#### 新要求

分类会话实例注入保持现有正例 / 反例分块机制，并额外加入最近 30 条实例。

分类会话中应形成四个独立实例块：

```text
最近 30 条正例
最近 30 条反例
检索得到的正例
检索得到的反例
```

说明：

```text
最近 30 条实例与检索实例允许重复出现。
不要因为重复就合并或去重。
重复出现可以强化近期关键样例的显著性。
正例 / 反例沿用系统现有定义。
```

#### 预期效果

近期样例和检索样例分别发挥作用。最近 30 条保证近期偏好覆盖，检索样例保证当前交易相关性。

---

### 2.5 无 userNote 的实例在学习会话和分类会话中均标记为弱证据

#### 当前问题

用户纠正分类时，可能没有填写 `userNote`。这类样例只能说明“用户最终把该条目归为某类”，但不能说明原因。

如果模型根据无备注纠错强行总结原因，可能生成伪规律。

#### 新要求

无 `userNote` 的实例需要在两个层面体现弱证据标记。

#### A. 学习会话 Prompt 中明确说明

学习 Prompt 需要告诉模型：

```text
没有 userNote 的实例属于弱证据。
它只能作为用户最终分类的事实证据。
模型不得为其编造用户意图。
模型不得仅凭单条无备注实例总结稳定规律。
```

#### B. 学习会话和分类会话注入实例时，在 userNote 字段中体现

当实例没有 `userNote` 时，注入上下文时应在 `userNote` 字段中写入类似标记：

```text
[弱证据] 用户未提供原因；仅可作为最终分类事实，不可用于归纳原因或稳定规则。
```

如果实例本身已有用户备注，则保留原备注，不覆盖。

#### 预期效果

无备注纠错仍可作为历史样例使用，但不会被模型误当成强解释证据。

---

### 2.6 学习 Prompt 增加约束：不得生成依赖未来 userNote 的规则

#### 当前问题

学习会话中，历史实例的 `userNote` 是可见的；但未来待分类账单通常没有用户事后备注。

因此，如果模型生成类似规则：

```text
如果用户备注包含“和女朋友”，则分类为正餐
如果用户注释说明是补差价，则归为某类
```

这类规则在未来分类阶段不可执行。

#### 新要求

学习系统提示词中加入约束：

```text
学习时应优先总结未来分类阶段可见的信息。
包括交易时间、金额、对手方、商户或收款方、账单来源、支付渠道、交易备注、历史分类结果等。
不得生成依赖未来用户纠错备注才能执行的规则。
userNote 只能用于理解历史纠错原因，不能被当作未来待分类条目的可见字段。
```

#### 预期效果

减少不可执行记忆，提高学习结果在真实分类阶段的可用性。

---

### 2.7 用户开启分类时，若存在未学习实例，则分类前强制学习

#### 当前问题

如果学习触发机制依赖固定阈值，例如累计 5 条才学习，少量但关键的纠错可能迟迟没有进入学习流程。用户再次开启分类时，系统仍然使用旧记忆，容易重复犯错。

#### 新要求

当用户开启分类流程时，系统先检查是否存在未学习实例。

若存在：

```text
先触发一次学习会话
学习完成后再开始分类
```

UI 和状态行为要求：

```text
用户手动开启分类引擎后，若检测到仍有未学习实例，立即复用已有顶部提示通道弹出：
“有尚未学习的实例，AI 正在自动学习”。
AI 引擎工作标识随用户分类指令直接开启。
学习阶段不使用“某天正在 AI 流光处理”的日级分类标识，因为此时尚未正式开始处理某天分类。
学习结束后，若学习成功并即将切入正式分类流程，再通过同一顶部提示通道弹出：
“已学习完成，开始进行分类”。
学习结束后，再进入正式分类流程。
```

补充说明：

```text
上述两条顶部提示都属于轻量状态提示，不下沉为首页中部情景提示卡。
前一条用于解释“为什么 AI 已亮起但没有任何日期进入彩色流光”。
后一条只在“学习成功且后续确实继续进入分类会话”时出现；若学习失败并中断分类，则不显示该完成提示。
```

#### 预期效果

确保用户已完成的纠错不会被长期搁置，同时避免 UI 上误导用户认为某天账单已经开始分类处理。

---

### 2.8 用 Prompt 先约束记忆条目类型：实体记忆与规律记忆不得混写

#### 当前问题

当前记忆库暂未结构化区分“实体记忆”和“规律记忆”。这会导致模型把具体实体经验和抽象规律混在同一条记忆中。

例如：

```text
水之源川菜多次为正餐，因此川菜馆等餐厅……
```

这类写法同时包含具体实体经验和泛化规律，容易导致过度概括。

#### 新要求

第一轮不改记忆库结构，先通过学习 Prompt 进行约束。

学习 Prompt 应要求模型在生成或修改记忆条目时，明确区分两类：

```text
实体记忆：针对具体商户、具体收款方、具体平台、具体地点、具体对象的经验。
规律记忆：跨多个实体成立的通用判断规则。
```

约束：

```text
一条记忆只能属于一种类型。
一条记忆不能同时包含实体经验和泛化规律。
如果既发现了实体经验，又发现了可泛化规律，应拆成两条记忆。
没有多个实体支持时，优先写成实体记忆，不要过早泛化。
```

#### 预期效果

在不重构记忆系统的前提下，先减少“具体案例被抽象稀释”的问题，为后续实体记忆 / 规律记忆分层打基础。

---

## 3. 下一阶段继续讨论项

### 3.1 分类输出字段扩展

当前分类输出主要包括：

```text
最终分类
20 字以内理由
```

该形式适合展示给用户，但不利于内部审计、复盘和后续学习。

下一阶段需要设计更完整的输出 schema，可能包括：

```text
置信度
不确定原因
使用了哪些历史实例
是否依赖弱证据
是否建议用户复核
分类依据类型
```

需要明确：

```text
哪些字段展示给用户
哪些字段仅内部保存
哪些字段进入后续学习
哪些字段供检查会话使用
```

---

### 3.2 轻量级分类后检查会话

分类后检查会话暂不进入第一轮实施，下一阶段单独讨论。

背景是：分类完成后，系统可以通过一个轻量会话检查明显不一致、可疑分类、低置信结果或理由冲突。但该能力依赖分类输出字段、审计字段和结果维护机制。

下一阶段需要明确：

```text
检查哪些问题
检查会话输入什么
输出什么字段
是否自动修正
是否只提示用户复核
如何避免增加过多 token 成本
```

---

### 3.3 同商户识别与同商户样例强制注入

同一商户历史样例通常对分类很有价值。理想情况下，如果当前交易命中历史商户，系统应强制注入相关样例。

但当前需要先确认字段可靠性：

```text
counterparty 是否等同于商户
counterparty 是否可能是支付平台、收款码名称、个人名或门店名
同一商户是否存在多个别名
是否需要模糊匹配或归一化
```

在字段可靠性未确认前，不贸然实现同商户强 pin。

---

### 3.4 每条记忆的证据维护机制

当前系统有版本快照机制，但尚未完整维护“每条记忆来自哪些实例”。

后续若要提高可审计性，需要为记忆维护来源信息：

```text
记忆内容
来源实例 ID
支持样本数量
是否来自 userNote
是否包含无备注弱证据
创建时间
最后更新时间
置信度或证据等级
```

该机制会影响记忆存储结构、学习输出格式、版本快照和回滚逻辑，因此下一阶段单独设计。

---

### 3.5 低置信分类审计队列

当分类结果置信度低、证据冲突或结果异常时，系统可以把这些条目加入待复核队列，而不是让用户全量检查。

该能力依赖分类输出字段扩展。如果当前模型不输出置信度、不确定原因和证据类型，就难以稳定维护低置信审计队列。

因此该项需等待分类输出 schema 确定后再实施。

---

### 3.6 普通模型 / 推理模型分流

长期来看，Moni 应尽量做到：

```text
简单样本使用低成本模型
困难样本使用推理模型
```

这样能在保证精度的同时控制成本。

但当前系统使用统一自实现 LLM client。模型分流需要设计：

```text
模型路由逻辑
困难样本判定条件
不同模型配置
日志记录
失败回退
成本统计
```

本阶段先统一使用推理模型，并验证 thinking 参数是否生效。等基础能力稳定后，再讨论成本优化和模型路由。

---

### 3.7 完整实体记忆 / 规律记忆分层

第一轮只通过 Prompt 约束实体记忆和规律记忆不要混写，不立即改存储结构。

完整分层仍需下一阶段设计，包括：

```text
实体记忆 schema
规律记忆 schema
学习输出格式
记忆检索逻辑
上下文注入逻辑
冲突处理
证据追踪
版本管理
```

这是高价值改造，但不适合作为第一轮低成本改动。

---

## 4. 本轮不处理项

以下内容本轮明确不处理：

```text
额外邻近交易上下文机制
高金额 / 目标类别纠错优先学习
有 userNote 后立即学习
同商户连续纠错立即学习
重试策略优化
完整 benchmark 平台
完整 agent memory 重构
知识图谱式记忆系统
普通模型 / 推理模型路由
低置信审计队列
```

其中，额外邻近交易上下文暂不处理，是因为当前分类流程已经采用 3 天一组的方式，将待分类内容直接放入上下文，已经包含一定范围内的相邻交易信息。

## 6. 第一轮改造后的预期变化

第一轮改造完成后，Moni 应具备以下变化：

1. 模型默认输出更稳定，同时保留设置页调参空间。
2. 推理模型和 thinking 参数是否真实生效得到确认。
3. 用户近期纠错会更快进入学习和分类上下文。
4. 分类上下文中同时保留“最近实例”和“检索实例”的作用，不因去重损失显著性。
5. 无备注纠错仍能参与学习和分类，但被明确降级为弱证据。
6. 学习会话更少生成依赖未来 userNote 的不可执行规则。
7. 少量关键纠错不会因为未达到阈值而被下一轮分类忽略。
8. 记忆条目在 Prompt 层面开始区分实体经验和泛化规律，减少混写和过度泛化。

第一轮目标不是完成完整 Agent Memory 架构，而是优先修复当前机制中最直接影响精度的低成本问题。
