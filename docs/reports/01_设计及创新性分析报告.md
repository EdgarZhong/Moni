# 设计及创新性分析报告

## Moni AI 自学习系统报告

## 摘要

本文聚焦 Moni 的 AI 自学习系统及其上下文工程实现，不再讨论比赛叙事、视觉设计或展示型内容。Moni 的核心价值，不在于一次性识别交易，而在于把用户的纠错、确认与手动录入持续转化为可复用的个性化知识，并在后续分类、学习、收编三个环节中反复注入。整个系统以本地优先的数据结构为基础，通过提示词拼接、实例库分层、记忆快照、版本控制和优先级仲裁，形成一套可持续演进的个性化分类闭环。

## 一、核心问题

个人财务分类最难的地方，不是“能不能分”，而是“能不能持续分对同一个人”。通用模型可以给出平均意义上的分类，但用户真正需要的是：

- 识别这个具体用户的消费语境
- 记住用户改过什么、确认过什么
- 在后续推理中优先遵循这些偏好
- 在规则变化或数据回滚时保持可恢复、可重建

Moni 的设计正是围绕这四个问题展开。

## 二、系统总览

当前代码里的自学习体系由四层构成：

1. 分类层：`SystemPrompt` 把 `category_list`、`reference_corrections`、`days` 和动态注入的 `self_description`、`memory` 拼成一次分类请求。
2. 学习层：`LearningSession` 读取实例库增量，生成 `ADD / MODIFY / DELETE` 记忆操作。
3. 收编层：`CompressionSession` 读取当前记忆和全量实例库，压缩冗余记忆并保留有效规则。
4. 仲裁层：`Arbiter` 保证用户手动修正永远高于 AI 自动结果，避免系统反复覆盖用户决策。

这四层并不是松散拼接，而是围绕“用户输入如何变成稳定上下文”这一条主线组织的。

## 三、当前实现的关键上下文

### 3.1 分类上下文

分类时，模型看到的不是原始流水，而是经过运行时组装的结构化上下文：

- `category_list`：账本当前有效的分类体系
- `reference_corrections`：实例库检索结果，分为 `misclassified_examples` 和 `confirmed_examples`
- `days`：按日期分批的待分类交易，当前实现最多 3 天，日期倒序
- `self_description`：来自全局自述文件的用户长期偏好
- `memory`：按账本保存的学习快照，承载抽象后的分类规律

这里最关键的是 `reference_corrections` 的双区块设计：

- `misclassified_examples` 保存 B 类样本，保留 AI 的错误判断与错误理由，并在注入时加 `[错误判断]` 前缀
- `confirmed_examples` 保存 A / C / D 类样本，承载用户确认过的正确答案

这使分类模型同时看到“正例”和“反例”，不是只看结果，而是看结果如何被纠正出来的。

### 3.2 学习上下文

学习层不是拿一份原始例子做总结，而是拿“相对上次学习基线的变更”做归纳。当前实现里，学习输入的核心字段是：

- `mode`: `delta` 或 `full_reconcile`
- `from_revision` / `to_revision`
- `upserts`
- `deletions`
- `current_examples`：仅在 `full_reconcile` 时注入全量现行实例库

这套设计的价值在于，学习模型始终知道：

- 哪些样本是新增或更新后的有效信号
- 哪些样本已经被删除
- 什么时候不能再相信增量日志，必须回退到全量重建

这比“只给一坨历史数据让模型自己猜”稳定得多。

### 3.3 收编上下文

收编层的作用不是再训练新的偏好，而是压缩已有记忆。当前实现会同时读取：

- 当前记忆快照
- 当前分类体系
- 当前实例库全量

然后判断是否触发收编，并把当前记忆压缩为更短但不丢失关键信息的编号列表。收编时必须保留仍有实例支撑的规则，不能只凭记忆文本本身删除或合并。

## 四、AI 自学习系统的创新点

### 4.1 不是“训练模型”，而是“训练上下文”

Moni 没有走微调模型的路线，而是把个性化知识拆成三类可控上下文：

- 自述：用户主动声明的长期偏好
- 实例：具体交易及其纠错轨迹
- 记忆：从实例中抽象出的稳定规则

这意味着系统的“学习”不是改模型参数，而是持续改进下一次推理时的输入质量。对移动端、本地优先和外部 LLM API 组合来说，这是更现实的路径。

### 4.2 正反例分离

B 类样本是系统最有价值的学习信号，因为它同时包含：

- AI 当时为什么会错
- 用户后来为什么改成别的类别

当前实现没有把错误样本简单当作普通正例处理，而是通过独立区块和 `[错误判断]` 前缀显式提示模型：这是一条要避免复现的错误路径。这一点对分类系统特别重要，因为很多误判不是“答案不知道”，而是“错误推理模式会重复出现”。

### 4.3 版本化与可恢复

实例库和记忆都不是不可变文本，而是带版本和变更基线的状态。这样做的意义有三个：

- 学习可以增量推进
- 失败可以回退重试
- 数据回滚后可以切换到 `full_reconcile` 重建

对于一个持续运行的个性化系统来说，这比一次性生成一份记忆更可靠。

### 4.4 用户主权优先

当前代码中，用户的手动决定始终高于 AI。这个优先级不是文案口径，而是系统行为边界：用户修正后，后续分类和学习都必须以用户修正为真。换句话说，AI 可以建议，但不能推翻用户。

## 五、与代码实现一致的结论

当前代码库里，AI 自学习系统已经形成了比较完整的闭环：

- 分类阶段用 `reference_corrections + self_description + memory` 做上下文注入
- 学习阶段用 `delta / full_reconcile` 保证记忆更新可控
- 收编阶段用全量实例库纠正和压缩记忆
- 仲裁阶段保证用户修正不会被后续 AI 结果覆盖

这套体系的核心创新，不在于某一个单点算法，而在于把“用户纠错”变成了“长期可累积的上下文资产”。它解决的是个性化分类系统最关键的问题：如何让模型持续理解同一个用户，并且在数据回滚、规则变化和多轮交互中保持稳定。

## 六、可直接对外表述的核心观点

- Moni 的本质不是一个记账 UI，而是一套可持续演进的个性化分类上下文系统。
- 它的学习对象不是原始分类结果，而是用户纠错后的稳定偏好。
- 它的技术重点不是模型微调，而是上下文工程、实例库分层、记忆快照和优先级仲裁。
- 它的关键创新是把“用户的每一次修正”变成“下一次推理可用的知识”。
- 它的系统价值在于，能在本地优先架构下实现真正的长期个性化，而不是一次性的自动分类。
