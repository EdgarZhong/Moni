# Moni AI 智能记账助手技术研究报告

**报告定位：** 这份报告补充 `docs/AI_SELF_LEARNING_DESIGN_v8.md` 没有展开的代码库上下文，重点描述“当前代码到底已经实现到哪里、各模块如何协作、测试如何验证、还有哪些点适合继续讨论和改进”。

---

## 摘要

Moni 的 AI 系统不是单点“自动分类器”，而是一套围绕用户纠错逐步演化的本地优先闭环：交易进入账本后，系统先把“哪些天需要 AI 重新处理”落成分类运行态，再由批处理引擎按天消费，结合用户自述、记忆快照、实例库参考案例和当前账本分类体系生成分类结果；结果写回后又会反向沉淀为实例库、记忆快照和新的自动学习窗口。设计文档 v8 主要描述这套系统应该遵守的原则，而这份报告则说明当前代码库已经把这些原则落实到了哪些类、哪些文件、哪些测试入口，以及哪些地方仍然保留了兼容口径或未来扩展位。

当前实现的技术重心可以概括为四层：

1. **运行态层**：`ClassifyIndex` / `ClassifyRuntimeStore` 维护按账本持久化的分类索引和运行态，`ClassifyTrigger` 负责入队与恢复。
2. **消费层**：`BatchProcessor` 按日期批量消费任务，`PromptBuilder` / `SystemPrompt` 负责把代码里的真实上下文拼成模型输入。
3. **学习层**：`ExampleStore`、`LearningSession`、`LearningAutomationService`、`MemoryManager`、`SnapshotManager` 共同完成从纠错样本到记忆快照的闭环。
4. **编排层**：`LedgerService` 与 `AppFacade` 把用户操作、后台学习、调试测试和 UI 读模型连接起来。

## 三、分类链路的当前实现

### 3.1 任务不是“队列”，而是“运行态索引”

代码里保留了 `ClassifyQueue.ts` 这个文件名，但它的主语义已经不是传统队列，而是运行态索引。`ClassifyRuntimeStore` 才是最终落盘容器，负责把每个账本的分类状态统一存进 `ledgers/{ledger}/classify_runtime.json`。

当前运行态字段如下：

- `version`：运行态版本。
- `revision`：索引版本号。
- `metrics`：包含 `emptyTaskConsumedCount` 和 `lastEmptyTaskDate`。
- `queue`：按时间排序的待消费日期视图。
- `dirty_count_by_date`：主状态，表示每个日期还有多少条“应重跑”的贡献。
- `needs_rebuild` / `rebuild_reason`：索引失真或旧数据迁移时的重建标记。
- `enqueue_recovery`：入队失败补偿。
- `confirm_recovery`：重分类确认补偿。

这意味着当前实现已经不再把“任务”与“恢复”分散在多个独立文件里，而是统一收口到一个运行态文件中。设计文档里强调“统一运行态”，代码里已经把这点做实了。

### 3.2 任务生产只负责“哪一天需要重跑”

当前任务生产来自三条路径：

1. **账单导入**：`ClassifyTrigger.enqueueCsvImport()` 只把“未锁定且仍为 `uncategorized`”的日期入队。
2. **用户确认重分类**：`ClassifyTrigger.enqueueConfirmedDates()` 只入队已经完成前置改写的脏日期。
3. **恢复重放**：`ClassifyTrigger.recoverPending()` 从 `enqueue_recovery` 回放失败日期。

值得注意的是，`ClassifyTrigger` 不负责生成分类结论，它只负责把需要 AI 重新处理的日期写入运行态。这个边界很重要，因为它把“数据生产”和“模型消费”拆成两件事，避免任何上游链路偷偷直接改写分类结果。

### 3.3 `BatchProcessor` 的消费口径

`BatchProcessor` 是当前 AI 消费的真正入口。它的关键约束不是“能不能跑”，而是“怎么保证只在当前范围内消费、只按有限批次消费、只读不写生产边界”。

当前实现有几个非常明确的限制：

- `DEFAULT_MAX_PARALLEL_DAYS = 3`，单次最多消费 3 天。
- `getConsumableTaskBatch()` 会先读取当前账本的消费范围，再过滤任务日期。
- 当前范围为空时，不会把范围外 backlog 一起拉进来。
- `buildPromptDayBatches()` 会把同一天的完整交易上下文一起带进 Prompt，而不是只喂单条记录。

这意味着批处理并不是“把队列清空”，而是“在可消费范围内按最近日期优先处理有限批次”。这和设计文档里的“最近日期优先”一致，也和当前首页的日期范围控制逻辑一致。

### 3.4 Prompt 的真实输入结构

`PromptBuilder.build()` 现在注入的不是纯文本拼接，而是一个结构化 JSON：

- `category_list`
- `reference_corrections`
- `days`

其中：

- `category_list` 来自当前账本的分类定义；
- `reference_corrections` 来自 `ExampleStore.retrieveRelevant()`；
- `days` 由 `BatchProcessor` 按批次组装，且保持“最近日期优先”的顺序。

`SystemPrompt` 进一步规定了模型行为：

- `self-description` 最高优先级；
- `reference corrections` 作为强锚点；
- `learned preferences` 作为稳定模式；
- `income` 默认规则与退款例外必须同时成立；
- `reasoning` 不能超过 20 个字；
- 输出必须是严格 JSON，不能夹杂解释性文字。

这部分是技术研究报告里必须补充的，因为设计文档只讲“应该注入哪些上下文”，而代码已经把“注入结构”和“约束词法”写成了可验证的实现。

### 3.5 结果写回不是直接落分类，而是经由领域仲裁

模型输出后，并不是直接写入账本，而是先经过 `Arbiter`。

`Arbiter` 当前的真实优先级是：

1. `USER`
2. `RULE_ENGINE`
3. `AI_AGENT`

这里有一个很重要的现状说明：`RULE_ENGINE` 已经存在于领域协议里，但它更多是预留的中间层，当前主流程还没有把它作为与用户、AI 同等频率参与写回的常驻来源。也就是说，代码层面已经给未来规则引擎留了位置，但现阶段真正主导结果的仍然是用户意图与 AI 提案。

这个结构的意义在于：即便 AI 重新跑一遍，已经由用户确认的分类也不应该被轻易覆盖。`LedgerService` 负责把最终裁决转成补丁，再把补丁写回账本并同步触发实例库和学习窗口。

## 四、自学习系统的当前实现

### 4.1 实例库不是“样本列表”，而是“版本化证据库”

`ExampleStore` 是当前自学习系统的核心事实源之一。它的主文件是 `examples.json`，并且同时维护 `example_changes.json` 作为变化日志。

当前实例条目字段包括：

- `id`
- `created_at`
- `time`
- `sourceType`
- `rawClass`
- `counterparty`
- `product`
- `amount`
- `direction`
- `paymentMethod`
- `transactionStatus`
- `remark`
- `category`
- `ai_category`
- `ai_reasoning`
- `user_note`
- `is_verified`

这组字段有一个明显特征：**它保存的是交易语义和分类证据，而不是单纯的“模型最终答案”**。这也是为什么当前系统可以在不改账本主结构的前提下持续演化学习逻辑。

### 4.2 A / B / C / D 的运行时判定

当前代码里，A / B / C / D 不是落盘字段，而是运行时推导结果。

`ExampleStore.resolveExampleKind()` 会根据当前条目的事实字段推断类型：

- **D**：`sourceType === 'manual'`，属于随手记样本。
- **B**：`ai_category` 存在且和 `category` 不同，表示 AI 分错、用户纠正。
- **A**：`ai_category` 存在且和 `category` 相同，表示 AI 分对、用户确认。
- **C**：`ai_category` 为空但用户已经给出了有效分类，表示 AI 没参与但用户直接归类。

这个设计和文档里的“学习实例分层”完全一致，但实现上更灵活：类型不是写死在存储层，而是根据当前事实推导。这样做的好处是存储稳定、策略可调，坏处是研究评估时必须明确知道推导规则，而不能只看文件字段。

### 4.3 变更日志与学习基线

`ExampleStore` 维护 `revision` 和 `example_changes.json`，这是当前增量学习的基线。

学习窗口的读取逻辑是：

- 如果 `currentRevision === lastLearnedRevision`，说明没有新增变更；
- 如果 `currentRevision < lastLearnedRevision`，说明 revision 回退或历史断裂；
- 如果 change log 不可信，就退化成 `full_reconcile`；
- 正常情况下使用 `delta` 模式，只学习净变更。

这部分对应 `LearningSession.buildLearningPayload()` 的两种模式：

- `delta`
- `full_reconcile`

`full_reconcile` 模式下，`current_examples` 会把当前全量实例库直接注入学习上下文，这意味着当前系统对“变更日志丢失、revision 不连续、历史窗口不可恢复”是有兜底的。

### 4.4 学习会话实际在学什么

`LearningSession` 不直接改规则，它做的是“把实例库变化转成记忆更新操作”。

当前学习会话输出的是这三种操作：

- `ADD`
- `MODIFY`
- `DELETE`

记忆内容本身是一个**编号列表**，每一行一条自然语言规则。`MemoryManager.applyOperations()` 会先按索引倒序执行删除和修改，再追加新增，最后写成新的快照。

这意味着当前记忆系统有两个特点：

1. 它是 AI 可读的自然语言规则列表，不是结构化数据库。
2. 它是写后快照模型，任何一次学习或用户编辑都会生成新的记忆历史版本。

这和设计文档强调的“AI 自由决定记忆内容”是吻合的，但从研究视角看也带来一个现实问题：**自然语言记忆非常灵活，但统计评估和结构化分析成本更高**。这一点后面会单独讨论。

### 4.5 自动学习的触发条件

`LearningAutomationService` 负责判断当前账本是否应该自动学习。

它的判定依赖三个事实：

1. `ai_prefs.json` 中的学习配置；
2. 当前实例库相对上次学习的净变更量；
3. 当前模型配置是否可用。

当前实现里还有一个很关键的并发保护：`inFlightLedgers`。当某个账本正在学习时，后续触发会被挡住，避免多个入口同时把同一账本塞进多个学习会话。

`LedgerService` 在以下行为后会调用自动学习评估：

- 用户修正分类；
- 用户修改 `user_note`；
- 用户锁定 / 解锁交易；
- 实例库被删除或更新。

这说明当前系统并不是只在“显式点击立即学习”时才会学习，而是把学习嵌进了常见的用户纠错流程。

### 4.6 收编机制是“压缩记忆”，不是“删旧样本”

`CompressionSession` 解决的是另一个问题：记忆快照会越来越长，Prompt 不能无限膨胀。

当前实现中：

- `shouldTrigger(currentCount, threshold)` 只在条目数超过阈值时触发；
- `computeTargetCount(currentCount, ratio)` 用 `floor(currentCount * ratio)` 计算目标上限；
- 收编时会把当前记忆和当前实例库全量一起送进模型；
- 收编输出必须是一份新的、压缩后的编号列表。

这里有一个重要的现状：**收编不是删除历史，而是让模型重写一版更短、更稳定的记忆**。这和学习会话相互补充：学习会话负责把新知识写进去，收编会话负责把旧知识整理好。

## 五、当前代码库里的关键上下文

这一节是设计文档里不会细讲、但技术研究报告必须交代清楚的代码上下文。

### 5.1 真实存储路径

当前正式运行时持久化已经统一写入 `Directory.Data`，对应的关键路径是：

- `ledgers.json`
- `secure_config.bin`
- `self_description.md`
- `ledgers/{ledger}/ledger.json`
- `ledgers/{ledger}/ai_prefs.json`
- `ledgers/{ledger}/budget.json`
- `ledgers/{ledger}/examples.json`
- `ledgers/{ledger}/example_changes.json`
- `ledgers/{ledger}/classify_runtime.json`
- `ledgers/{ledger}/memory/index.json`
- `ledgers/{ledger}/memory/{snapshot_id}.md`

这里有一个很重要的实现事实：**`self_description.md` 是全局文件，不是账本内文件**。它的优先级在 `SystemPrompt` 里也是最高的，代码已经把“全局自述”和“账本记忆”拆成两层了。

### 5.2 账本级 AI 配置的当前默认值

`ai_prefs.json` 由 [`LedgerPreferencesManager`](../../src/logic/application/services/LedgerPreferencesManager.ts) 管理，当前默认值是：

- 学习阈值 `5`
- 自动学习开启 `true`
- 收编阈值 `30`
- 收编压缩比例 `0.7`

这四个值已经在代码中冻结，并且和设计文档保持一致。技术研究上比较重要的一点是：它们不是散落在多个服务常量里，而是统一由账本级配置管理器读写。

### 5.3 记忆快照的实际行为

`MemoryManager` 和 `SnapshotManager` 共同决定记忆的生命周期。

当前行为是：

- 记忆不是直接改一个固定文件，而是每次保存都生成新快照；
- `current_snapshot_id` 指向当前生效快照；
- 快照是有序列表，每行一条；
- 用户编辑、AI 学习、AI 收编都会生成新的历史版本；
- 系统会保留最近 30 个快照，删除最旧且非当前的版本。

这意味着记忆系统的历史是可追溯的，但也是“写后快照式”的。对于研究和讨论来说，这很关键，因为它决定了我们可以如何回溯某次学习或某次用户手改后的行为变化。

### 5.4 UI 和业务的边界

`AppFacade` 是当前表现层和业务层之间的唯一硬边界。

表现层能拿到的是读模型，比如首页数据、设置页数据、分类索引状态、学习状态、收编上下文；不能直接持有服务层内部状态。

这有两个直接结果：

1. UI 切换时不会自己重新造一套分类或学习逻辑。
2. 研究和测试可以围绕 facade 的读模型做断言，而不是从 UI 组件里猜状态。

这也是为什么当前调试入口里可以直接暴露 `window.__MONI_DEBUG__` 和 `window.__MONI_E2E__`：它们本质上是在验证 facade 暴露出来的稳定业务事实，而不是在验证某个页面局部状态。

## 六、测试与研究验证入口

当前仓库里，AI 系统相关的验证不只是“看页面有没有显示”，而是有一组专门的调试与测试入口。

### 6.1 浏览器调试入口

当前开发态稳定入口包括：

- `window.__MONI_DEBUG__`
- `window.__MONI_E2E__`

其中，和 AI 系统直接相关的调试能力包括：

- `classify.getQueue()`
- `classify.getIndex()`
- `classify.enqueueDate()`
- `learning.getDeltaPayload()`
- `learning.getAutoTriggerState()`
- `compression.getContext()`
- `compression.run()`

这些接口使得我们可以直接研究“当前系统是如何构造学习窗口和收编上下文的”，而不是只在 UI 上看一个抽象结果。

### 6.2 现有自动化测试

当前代码库已经提供了几类很有针对性的研究测试：

- `runExampleStoreSpecTest`
- `runLearningPayloadSpecTest`
- `runLearningAutomationSpecTest`
- `runCompressionSpecTest`
- `runClassifyLockBoundaryTest`
- `runClassifyIndexIncrementalTest`

这些测试并不是泛泛的“有一个测试集”，而是直接对应当前实现中的关键风险点。

举几个例子：

- `runLearningAutomationSpecTest` 会验证学习阈值、`autoLearn` 开关和 pendingCount 的关系。
- `runCompressionSpecTest` 会验证收编阈值、压缩比例和 targetCount 计算。
- `runClassifyLockBoundaryTest` 会验证锁定交易不会错误地产生脏日期。
- `runClassifyIndexIncrementalTest` 会验证同日多条脏记录的聚合计数、状态变化是否影响 dirty 判定、以及重分类只减掉自己的贡献。

这些测试让我们能从“结果表现”反推“系统规则”，这对于后续改进非常重要。

### 6.3 当前还缺少的研究指标

虽然有这些测试入口，但从研究角度看，当前系统还缺少更明确的量化指标，例如：

- 学习后分类准确率如何稳定提升；
- 收编前后 Prompt 长度如何变化；
- 不同商户 / 不同场景的误分类类型分布；
- 规则更新与用户修正之间的冲突比例；
- 自动学习触发的收益是否足以覆盖 LLM 调用成本。

这部分是这份研究报告要主动提出的问题，而不是装作已经解决。

## 七、当前实现的优点与讨论点(小型AI生成建议，仅供参考)

### 7.1 已经比较稳的部分

1. **运行态与学习态分离清晰**  
   分类运行态、实例库、记忆快照、账本正文和全局自述已经各自有明确文件边界。

2. **学习窗口是版本化的**  
   `revision` + `example_changes.json` 让系统可以区分增量学习和全量回退。

3. **UI 与业务边界明确**  
   `AppFacade` 让页面只消费读模型，不直接碰内部状态。

4. **测试可以直接读业务事实**  
   调试入口和 E2E 入口已经能覆盖学习 payload、自动触发、收编、分类索引和锁定边界。

5. **模型提示词是结构化的**  
   Prompt 不是随机拼接文本，而是有明确 JSON 结构和系统提示约束。

### 7.2 适合继续讨论的部分

1. **记忆内容仍然是自然语言列表**  
   这让 AI 更自由，但也让评估更难。后续可能需要引入更明确的标签或分层摘要，以便做研究统计。

2. **`RULE_ENGINE` 仍是预留层**  
   当前已经有优先级位置，但还没有形成真正常驻的规则学习 / 规则写回体系。它适合做下一阶段研究。

3. **实例检索仍然主要靠启发式匹配**  
   商户匹配、关键词重叠、金额接近度、餐时匹配这四个维度是有效的，但权重如何调，仍然值得做实验。

4. **收编和学习都依赖 LLM 的自然语言输出**  
   这让系统灵活，但也带来输出稳定性问题。我们需要更好的失败分析和一致性评估。

5. **当前报告里最值得补充的是“实验记录”**  
   未来最好把每次测试里观察到的现象记录下来，比如某类误分类在学习后是否减少、收编前后 Prompt 是否稳定、自动学习触发是否真的减少人工干预。

## 八、结论

Moni 当前的 AI 系统已经不是“一个分类 Prompt”或者“一个记忆列表”，而是一整套跨层闭环：运行态索引负责收集待处理日期，批处理引擎负责消费，Prompt 层负责把上下文结构化，Arbiter 负责守住用户优先，ExampleStore / LearningSession / MemoryManager / SnapshotManager 负责把用户修正变成可持续演化的记忆，LearningAutomationService 和 CompressionSession 则负责让这套系统能长期跑下去。

如果只看设计文档，我们会知道系统应该做到什么；如果只看代码，我们会知道系统到底已经做到什么。这份技术研究报告的价值就在于把两者连起来：既保留设计目标，也保留当前实现的真实上下文，方便后续围绕“如何改进这套 AI 分类与自学习系统”展开讨论、实验和验证。
