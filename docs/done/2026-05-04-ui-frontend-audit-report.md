# 2026-05-04 表现层前端实现审查报告

## 审查范围

- 审查依据：`vercel-react-best-practices`
- 审查目标：结合本轮 UI bug 修复，复核当前主仓库表现层实现是否存在明显的重渲染、资源重复加载、首屏 bundle 过重、交互状态组织不稳等问题
- 重点文件：
  - `src/bootstrap/AppRoot.tsx`
  - `src/ui/pages/MoniHome.tsx`
  - `src/ui/pages/MoniEntry.tsx`
  - `src/ui/pages/MoniSettings.tsx`
  - `src/ui/features/moni-home/components.tsx`
  - `src/ui/hooks/useAiEngineControl.ts`
  - `src/ui/hooks/useMoniHomeData.ts`

## 总体结论

如果把关注点放在“用户是否会明显感到别扭、误触、跳变、状态不对劲”，那么当前表现层最值得继续盯住的不是纯性能数字，而是下面 4 类问题：

1. 交互状态和提交状态边界容易再次混掉，导致“点一下就生效”“取消按钮失真”这类问题回归
2. 单组件承载的交互职责过多，容易在无关状态变化时引发布局抖动、按钮跳位、面板重排
3. 跨页面 UI 状态读取链路过重，容易带来底部导航或控制态的无关刷新与视觉不稳定
4. 组件文件职责过于混杂，后续再修 bug 时很容易出现“改一处，抖另一处”的误伤

结论口径：**本轮 bug 修复本身已经把最明显的行为错误收住了，但从用户体验稳定性角度看，表现层还没有完全进入“放心继续堆交互”的状态。**

## 已确认的正向改进

### 1. Date Range Picker 交互语义已从“即时生效”修正为“草稿态 + 确认提交”

- `MoniHome` 现在把已提交态和弹窗草稿态拆开，避免快捷按钮一点击就立刻影响首页数据
- `DateRangeDialog` 的轨道边界固定使用账本真实数据范围，不再把 `MIN/MAX` 扩展到系统当前日期
- “无交集”场景已通过 `isEmpty` 显式传递给过滤链路，不再错误折叠成账本边界日

这是一次正确的状态建模收口，符合“交互中的暂存态不要直接污染最终提交态”的原则。

### 2. 快捷按钮换行抖动已被结构性修复

- 快捷按钮区从内容宽度驱动的 `flex-wrap` 改成稳定网格
- 激活态不再通过去掉边框等方式改变布局占位

这次修复方向正确，本质上是把问题从“靠运气的样式平衡”改成“稳定槽位布局”。

### 3. 返回键栈和覆盖层链路比此前更健康

- `MoniEntry` 现在只在覆盖层存在时注册返回键处理
- 密码页与表单覆盖层的优先级处理清晰

这一点对 Android 真机可用性很重要，当前实现方向是合理的。

## 当前问题

### P1. 交互状态与最终提交状态仍然需要继续冻结边界，避免再次出现“选了就生效”的错误绑定

- 位置：
  - `src/ui/pages/MoniHome.tsx`
  - `src/ui/features/moni-home/components.tsx`
- 现状：
  - `DateRangePicker` 这次已经修成“草稿态 + 确认提交”
  - 但它之所以此前连续出错，根因不是单个 if 写错，而是这条链路里同时存在 `requested / visual / applied` 三层语义
  - 这类组件一旦再次被顺手改动，很容易重新把“面板里正在选什么”和“页面当前已生效什么”混在一起
- 风险：
  - `取消` 重新失效
  - 快捷按钮再次变成立即提交
  - 无交集场景又被偷偷折叠成边界日
- 对用户的实际伤害：
  - 用户会直接怀疑“这个面板是假的”
  - 对筛选结果失去信任
- 建议：
  - 继续把这套三层语义明确固定在注释和接口命名里
  - 后续若再动日期范围相关逻辑，优先改适配层，不要直接在页面里混改

### P1. `MoniHome` 单组件承载的交互职责过多，容易引发无关抖动、跳位和错误重排

- 位置：
  - `src/ui/pages/MoniHome.tsx`
  - `src/ui/features/moni-home/components.tsx`
- 现状：
  - `MoniHome` 同时管理日期范围、看板轮播、趋势拖拽、分类过滤、拖拽细则、详情页、账本切换、提示卡、展开日卡等大量交互
  - 这次你看到的快捷按钮换行抖动，本质上就属于“无关状态变化把局部布局也一起扰动了”
  - 目前虽然已经把该问题修掉，但根部问题仍是：首页主容器承担的责任太多
- 风险：
  - 今天修的是按钮换行，明天可能变成卡片高度轻微跳一下
  - 今天修的是范围面板，明天可能是提示卡关闭时列表位置闪一下
  - 容易出现“问题不在当前功能，但每次都会被顺带碰到”
- 对用户的实际伤害：
  - 不是纯性能问题，而是明显可见的“不稳”
  - 用户会感到页面“发虚”“会跳”“不可信”
- 建议：
  - 优先按交互域拆分，而不是按“文件太长”机械拆分
  - 建议至少拆出：
    - 日期范围面板状态
    - 首页列表区
    - 拖拽细则 / 详情态

### P1. 跨页面 UI 状态读取链路过重，容易带来无关刷新和控制态不稳

- 位置：
  - `src/bootstrap/AppRoot.tsx`
  - `src/ui/hooks/useAiEngineControl.ts`
- 现状：
  - 根层 `BottomNav` 的 AI 控件状态，目前仍依赖 `getMoniHomeReadModel()`
  - 这意味着它拿的不是“一个轻量 UI 控件状态”，而是“整份首页读模型里顺带带出来的 UI 状态”
- 风险：
  - 底部导航的状态来源和首页读模型绑定过深
  - 后续如果首页读模型再变化，可能出现无关页面里的控制态瞬跳、闪动或延迟更新
- 对用户的实际伤害：
  - 用户会把它感知成“底部那个 AI 控件状态不稳”
  - 这类问题比纯慢一点更糟，因为它看起来像假状态
- 建议：
  - 给 AI 控件单独一条轻量状态链路
  - 不要再让根层常驻导航依赖首页整份读模型

### P2. 静态资源和页面级样式的注入位置不够稳定，容易制造首屏或切页时的视觉噪音

- 位置：
  - `src/ui/pages/MoniHome.tsx`
  - `src/ui/pages/MoniEntry.tsx`
  - `src/ui/pages/MoniSettings.tsx`
- 现状：
  - 三个一级页面都各自注入同一条字体 `<link>`
  - 页面内部也各自持有一部分 `<style>`
- 风险：
  - 这不一定立刻表现成性能问题，但很容易表现成字体、字重、布局的轻微抖动或初始化不一致
  - 尤其在切页、热更新、某些开发态环境下，这类问题更容易被放大
- 对用户的实际伤害：
  - 用户不会说“你这里静态资源没有上提”
  - 用户只会说“怎么一切页有点抖”“字怎么突然变了一下”
- 建议：
  - 字体资源尽量统一上提到入口
  - 页面内 `<style>` 只保留确实必须跟组件同生同灭的那一小部分

### P2. `components.tsx` 职责过于混杂，后续容易继续出现误改和连带问题

- 位置：
  - `src/ui/features/moni-home/components.tsx`
  - `src/bootstrap/AppRoot.tsx`
- 现状：
  - `BottomNav`、`DateRangeDialog`、`DayCard`、`OverviewCard`、`DisplayBoard` 都塞在一起
  - 全局导航、首页弹窗、首页卡片、首页图表这些并不是一个修改频率，也不是一个风险域
- 风险：
  - 改一个首页弹窗时，容易顺手碰到全局导航
  - 复审时不容易快速判断“这次动的到底是哪个交互域”
- 对用户的实际伤害：
  - 这类问题最终会转化成“修一个抖一个”
  - 用户会持续看到本不该互相影响的地方被带坏
- 建议：
  - 至少把 `BottomNav` 从首页组件集里独立出来
  - 日期范围弹窗和首页列表卡片也应拆成不同职责文件

## 建议项

以下建议按“先稳用户体验，再谈更深层优化”排序：

### 第一优先级

- 冻结 `DateRangePicker` 的三层语义边界，避免下轮再回归
- 把 `MoniHome` 按交互域拆开，优先减少“无关状态引发局部抖动”
- 给根层 AI 控件独立状态链路，减少跨页 UI 状态不稳

### 第二优先级

- 把字体和稳定样式上提，减少首屏与切页视觉噪音
- 把 `BottomNav` 从首页组件总文件里拆出，降低误伤范围

### 第三优先级

- 再考虑长列表渲染、memo 边界、资源加载等更偏工程优化的话题

## 与本轮 bug 修复的关系

本轮修复本身并没有偏离 `vercel-react-best-practices` 的方向，反而在两个点上是加分项：

1. `DateRangePicker` 改成草稿态后，减少了不必要的即时写回和状态串扰
2. 快捷按钮改成稳定网格后，避免了“视觉状态切换导致布局重排”的问题

但也正因为这轮修复把 `MoniHome` 的交互状态继续堆在单组件里，才更暴露出首页主容器已经接近“体验稳定性临界点”。如果后面还要继续加首页交互，建议优先做交互域拆分，而不是继续把逻辑堆回 `MoniHome.tsx`。

## 建议优先级

### 第一优先级

- 守住日期范围选择器的提交边界
- 拆开 `MoniHome` 的主要交互域
- 独立根层 AI 控件状态

### 第二优先级

- 上提字体和稳定样式
- 拆出 `BottomNav`

### 第三优先级

- 再做更偏工程层的渲染和加载优化

## 本轮完成度判断

- 若问题定义是“本轮 UI bug 是否已经修完”：**可以判定为已完成**
- 若问题定义是“表现层前端是否已经达到较稳定、较不容易继续抖动和误绑定的状态”：**还没有**

当前更准确的状态应为：

> 行为 bug 已收口；下一轮重点应转向“把用户能感知到的不稳因素继续拆掉”。
