# Moni 第二轮自学习系统改进变更 v1.2

> 本轮目标：在第一轮（`自学习系统改进变更_v0.3.md`）已建成的“上下文工程 + 推理参数 + 弱证据标记”基础上，做两件事——把模型已有的不确定性信号导出为结构化字段，把实例库检索从纯字符串匹配 + 经验权重升级为带预处理 / 质量分层 / 字符相似度 / 非语义字段加权的本地传统检索方案。记忆层的完整重构（实体/规律分层 + 变更可解释性 + 样本溯源）统一推迟到第三轮一次性进行，本轮不做半步改造。
>
> v1.2 在 v1.1 基础上对 §2.3 做实质性扩展：原 §2.3 仅冻结审计队列的过滤 / 排序 / 出列三项接口语义，并把 UI 推迟到下一轮；本版本随请教页（Inquiry Page，审计页改名）表现层规格同步推进，将 §2.3 扩展为完整的视图组织接口语义——按天聚合、filter 联动、会话视图快照、空状态分类等。本版本不修改 §2.1 / §2.2 任何内容。

## 1. 背景与本轮目标

第一轮 `自学习系统改进变更_v0.3.md` 已完成 12 项低成本精度优化并随 release `0.4.0` / `0.4.1` 收口（详见 `CLAUDE.md` Release Changelog）。第一轮上线后的真实运行日志显示：

- 推理模型链路（DeepSeek `thinking + reasoning_effort`、SiliconFlow `enable_thinking + thinking_budget`）已确认生效
- 分类精度显著提升，模型已能区分单人/双人正餐、严格执行金额阈值、综合相邻交易判断
- 但分类输出仍为 `{ id, category, reasoning(≤20字) }` 三字段，模型在内部推理中暴露的不确定性被压扁为短理由
- 实例库检索仍为纯字符串匹配 + 权重打分（4 条手写规则），技术含量偏低，无法处理"杨国福" vs "杨国福麻辣烫"、"美团" vs "美团下的具体商户"等场景
- 编码 agent 完成的三项前置调查（非语义字段同类条件概率、序列号 / 占位污染模式扫描、counterparty 字段可靠性研究）已为本轮检索升级提供量化基础

本轮的核心定位：

1. **把模型内部已存在的不确定性信号导出为结构化字段**，让后续审计、复核、模型路由都有稳定接口
2. **把实例库检索从经验权重 + 纯字符串匹配升级为本地传统 NLP 检索方案**：预处理剥离序列号污染 + counterparty 质量分层 + 字符 bigram 相似度 + 非语义字段加权，并以 agent 实测 lift 反推权重
3. **（v1.2 追加）把审计队列从纯接口冻结扩展到完整的视图组织接口语义**：随请教页表现层规格同步推进 §2.3 的接口扩展，避免表现层规格悬空在接口不完整的状态上

本轮验收标准是"字段能稳定输出、能进日志、能驱动审计；检索能命中之前匹配不上的同实体多写法与平台遮蔽案例；审计视图接口语义可被请教页规格直接消费"。

**关于记忆层**：本轮原稿曾包含"学习会话变更可解释性扩展"（每条 operation 附带 comment + 全局 diagnostic）作为部分覆盖 v0.3 §3.4 的过渡步骤。评估后认为：

- 现有记忆是纯文本 numbered list，加 comment 字段不够优雅（需要快照索引/正文分离）
- 半步改造会引入临时数据结构，下一步还要重构

因此**记忆层（实体/规律分层 + 学习变更可解释性 + 样本溯源）统一推迟到第三轮一次性进行**，本轮不做。

**关于 embedding 路线的回撤**：v1.1 初稿曾选定"API embedding（路径 B）+ JSONL sidecar 存储 + 后端写死模型"作为检索升级架构。再评估后判断该方案对 Moni 当前问题域是过度工程，本轮 §2.2 改用本地传统方法。回撤理由详见 §2.2 设计前提调整段。

## 2. 本轮确认实施项

### 2.1 分类输出 schema 扩展

#### 当前问题

当前分类输出 schema 仅含 `id / category / reasoning`，其中 `reasoning` 限制在 20 字以内。模型在内部推理（reasoning tokens）中已经能识别出"金额边界"、"邻近交易未确认"、"商户首次出现"等不确定信号，但这些信号被压缩成短理由后无法被后续模块消费，也无法做相关性分析。

#### 新要求

分类输出 schema 扩展为：

```json
{
  "results": [
    {
      "id": "transaction_id",
      "category": "category_key",
      "reasoning": "≤20字解释",
      "confidence": "high | medium | low",
      "uncertaintyReason": "≤60字；high 时必须为空字符串",
      "usedWeakEvidence": true,
      "evidenceIds": ["最多 3 个，必须来自 reference_corrections 或 days[] 中的 id"]
    }
  ]
}
```

#### A. confidence 三级语义

不做 0–100 打分。模型自报概率不校准；三级在样本量上来后能稳定算"high 命中率 / medium 命中率 / low 命中率"，0–100 的额外精度是噪音。

| 级别 | 语义 | 触发场景 |
|------|------|---------|
| `high` | 依据明确，记忆/实例/规则一致指向同一答案 | 同商户多次同类历史命中、`self_description` 直接对应、exact-ID 锚点命中 |
| `medium` | 有依据但存在缺口、金额/场景边界、邻近交易未完全确认 | 商户名相近但不完全一致、金额接近阈值、相关实例 `user_note` 解释不清 |
| `low` | 依据不足、正反例冲突、主要靠推理或 `[弱证据]` | 新商户首现、正反例并存、仅有 `[弱证据]` 实例支撑 |

**禁止 `high` 的硬约束**（写入分类 System Prompt）：

```text
以下场景禁止输出 confidence = "high"：
1. 当前交易商户在 reference_corrections 中没有任何历史命中
2. 仅依赖泛化规则推理，没有具体实例或自述支撑
3. 仅依靠 [弱证据] 实例
4. 检索命中的正反例之间存在矛盾
满足以上任一条件时，至少为 "medium"。
```

#### B. uncertaintyReason

- `confidence === "high"` 时**必须**为 `""`（空字符串），写入 Prompt 强约束
- `confidence === "medium" / "low"` 时**必须**给出 ≤ 60 字的不确定来源
- 与 `reasoning` 分工明确：`reasoning` 是"为什么是这个分类"（用户可见，正面表述），`uncertaintyReason` 是"哪里不确定"（仅审计/日志可见，负面表述）

**示例**：

```text
reasoning = "正餐时段川菜消费"
uncertaintyReason = "金额28元偏低，可能是单人正餐"
```

#### C. usedWeakEvidence

判定规则写入分类 System Prompt：

```text
若本次分类引用的 reference_corrections 中，任何一条的 user_note 以 "[弱证据]" 开头，
则 usedWeakEvidence = true；否则为 false。
```

代码侧后置校验：根据 `evidenceIds` 反查实例库的 `user_note` 字段确认。模型与代码不一致时记 warning 日志。

#### D. evidenceIds

| 维度 | 约束 |
|------|------|
| 数量 | 上限 3 个；可以为空数组（仅推理得出的分类） |
| 来源 | **必须**来自本次注入的 `reference_corrections`（任一区块）或 `days[]` 的同一交易（exact-ID 锚点）|
| 防幻觉 | 后处理校验：剔除不在注入实例 ID 集合中的 ID；剔除后若 `confidence === "high"` 但 `evidenceIds.length === 0`，强制降级为 `medium` 并记 warning 日志 |
| 语义 | "起决定作用的样本"，不是"看过的样本" |

#### E. 字段去向

| 字段 | 用户可见 | LLM 日志 | 写入 transaction | 写入实例库 | 进入学习/收编 prompt | 驱动审计 |
|------|---------|---------|----------------|-----------|--------------------|--------|
| `category` | ✓ | ✓ | ✓（`ai_category`） | ✓ | ✓ | — |
| `reasoning` | ✓ | ✓ | ✓（`ai_reasoning`） | ✓ | — | — |
| `confidence` | 仅审计 UI（下轮） | ✓ | ✓（新增 `ai_confidence`） | ✗ | ✗ | ✓（派生 `ai_needs_review`） |
| `uncertaintyReason` | 仅审计 UI（下轮） | ✓ | ✓（新增 `ai_uncertainty_reason`） | ✗ | ✗ | ✓ |
| `usedWeakEvidence` | — | ✓ | ✓（新增 `ai_used_weak_evidence`） | ✗ | ✗ | 触发因子 |
| `evidenceIds` | 仅审计 UI（下轮） | ✓ | ✓（新增 `ai_evidence_ids`） | ✗ | ✗ | — |

**关键边界**：

- **实例库 schema 不动**（除 §2.2 派生的字符串预处理字段外）。实例库是学习的输入，只承载用户认可/否决的事实信号，不混入模型自报的元认知
- **学习/收编 Prompt 不消费这些新字段**。学习仍然只看实例库，不看 transaction 上的 AI 自报字段
- 这些字段的下游消费者是 LLM 日志（本轮）+ 审计 UI（下一轮）+ 模型路由分析（再下一轮）

#### F. 派生字段

| 字段 | 派生规则 | 写入位置 |
|------|---------|--------|
| `ai_needs_review` | `confidence === "low"` 或 `(confidence === "medium" && amount >= NEEDS_REVIEW_AMOUNT_THRESHOLD)` 或 同商户历史分类不一致 或 `counterparty` 在实例库首次出现且 `confidence !== "high"` | 代码后处理后写入 transaction |
| `ai_evidence_type` | 由 `evidenceIds` 反推：命中商户名完全一致 → `entity_match`；命中泛化规则但无实例 → `pattern_match`；`evidenceIds` 为空 → `pure_inference` | LLM 日志（本轮不入 transaction）|

`NEEDS_REVIEW_AMOUNT_THRESHOLD` 默认值与设置入口暂不暴露，先用代码层常量 `100`（元），跑 1–2 轮真实数据后再调。

#### 预期效果

- 模型已有的不确定性信号被结构化导出，可入日志、可分析、可驱动审计
- 后续审计 UI、模型路由分析、学习诊断均有稳定接口
- 输出 token 略增（每条交易增加约 100–200 字 JSON），相对 reasoning token 占比可忽略

### 2.2 实例检索 / 评分注入升级

#### 当前问题

当前实例库检索（`AI_SELF_LEARNING_DESIGN_v8.md` §2.1.2）采用纯字符串匹配 + 经验权重打分：商户名匹配 > 品类相似 > 金额区间 > 时段相近。技术含量偏低，存在以下未覆盖场景：

- 商户名近义不同写：`杨国福` vs `杨国福麻辣烫`、`西安唐久便利连锁有限公司` vs `唐久便利西安`
- 平台名遮蔽真实商户：`美团` / `淘宝闪购` 等 counterparty 实际指向 product 中的真实商户
- 校园 / 机构聚合入口：`西北工业大学`、`扫码点单店主` 等 counterparty 不代表实体
- 商户名带序列号 / 订单号尾缀（如 `一卡通支付_325`），字符串匹配把同实体不同次记录拆散
- 经验权重未经数据验证，金额/时段/方向/支付方式各自鉴别力不清

编码 agent 已完成的三项前置调查给出了量化结论：

| 任务 | 关键发现 |
|------|---------|
| 任务一：非语义字段同类条件概率 | 金额 ±10% lift 1.91（alipay）/ 1.95（wechat）；小时同 lift 1.44 / 1.82；weekday / direction lift ≤ 1.06 几乎无信号；paymentMethod 对 wechat 有用（lift 1.49）对 alipay 几乎无用（lift 1.06） |
| 任务二：序列号 / 占位污染识别 | underscore_numeric_suffix（67 次）、long_digit_run（53 次）、KFC_PRE（21 次）等模式高度规则化，可正则确定性剥离 |
| 任务三：counterparty 字段可靠性 | 35.5% 记录的 counterparty 不是真实商户（平台 / 泛化机构 / 占位）；同实体多写法仅 9.3%，且都是字符高重叠案例 |

#### 设计前提调整：放弃 embedding 路线，采用本地传统方法

v1.0 初稿曾选定"API embedding（路径 B）+ JSONL sidecar 存储 + 后端写死模型"。再评估后回撤，原因：

**embedding 方案的代价**：

- 网络依赖：每次新增实例都要调用 embedding API；离线场景检索能力降级
- 复杂度：API Key 管理、模型版本绑定、向量 sidecar 文件、配额/失败重试、降级回退
- 不确定收益：实例库规模小（当前 18 条，预期生命周期上限约 500 条），向量检索的相对优势主要体现在大规模 + 强语义近义场景

**本地传统方法已足够**

#### A. 序列号 / 占位污染预处理

基于 agent 任务二输出的污染模式，在所有相似度计算前先做规则化清洗。

**清洗规则集（按优先级顺序应用）**：

| 优先级 | 模式名 | 正则 | 替换策略 | 任务二实测 |
|------:|-------|------|---------|----------|
| 1 | replacement_char_encoding | `[�]+` | 删除 | 1 次 / wechat |
| 2 | payment_code_verbose | `支付码:\d+` | 删除 | 13 次 / alipay |
| 3 | kfc_pre_order | `KFC_[A-Z0-9_]*\d{12,}` | 替换为 `KFC` | 21 次 |
| 4 | app_order_long_tail | `(?:美团\|大众点评)App-\d{12,}` | 替换为对应平台名 | 13 次 / alipay |
| 5 | cn_order_long_number | `订单[-_:\s]*\d{6,}` | 删除尾部数字 | 3 次 |
| 6 | hyphen_long_numeric_tail | `-\d{8,}$` | 删除 | 16 次 |
| 7 | parenthesized_long_number | `\(\d{8,}\)` | 删除 | 13 次 / alipay |
| 8 | long_digit_run | `\d{10,}` | 删除 | 60 次 |
| 9 | underscore_numeric_suffix | `_\d+$` | 删除 | 67 次 |
| 10 | phone_number | `(?<!\d)1[3-9]\d{9}(?!\d)` | 删除 | 7 次 / wechat |
| 11 | timestamp_embedded | `消费时间:\d{4}-\d{2}-\d{2}\s+\d{2}:\d{2}:\d{2}` | 删除 | 5 次 / alipay |

清洗规则应用于 `counterparty` 和 `product` 两个字段，产出 `counterparty_normalized` / `product_normalized` 派生字段。**原始字段在实例库和 transaction 上保持不变**；归一化字段仅用于相似度计算与质量标记。

清洗后再做基础归一化：全/半角统一、英文 case-insensitive、首尾空白去除、连续空白合并为单空格。

> 用户注：后续整套检索-评分系统NLP框架待谨慎设计，服务两个下游目标：1. 服务于实例注入的上下文重组织，以实现鲁棒的相关实例召回和字段级参考指导，提升分类精度，降低智能要求；2. 服务于模型路由，合理评估待分类条目的歧义度和分类难易程度。

### 2.3 低置信审计队列接口 / 视图组织语义（v1.2 扩展）

#### 当前问题

v1.1 §2.3 原本只冻结了"过滤 + 排序 + 出列"三项接口语义，定位为"纯运行时视图，不引入新文件、不引入新字段"，UI 形态推迟到下一轮专项。

随请教页（Inquiry Page，审计页改名）表现层规格同步推进，本节扩展接口语义以承接表现层的视图组织需求。**本扩展不引入新的持久化文件、不引入新的 transaction 字段**，仍保持 v1.1 §2.3 的"零迁移"原则；扩展的内容是**接口返回结构**、**排序与可见性的派生规则**、**会话视图快照的语义边界**、**空状态的接口分类**。

#### 范围调整说明

v1.1 §2.3 的"UI 推迟说明"小节由 v1.2 的 §2.3-G 替代——表现层规格已落入 `docs/design/spec/SPEC_Inquiry_Page.md`，本节只冻结接口与语义。

以下小节按 A → G 顺序定义。

#### A. 数据接口：返回结构（v1.1 §2.3-A 扩展）

接口返回两类条目，**统一按天组织**：

```text
返回的"天"集合定义为：
  天 D 进入返回集合 ⟺ 当天存在至少一个满足下列条件的条目：
    ai_needs_review === true && is_verified === false

每个进入返回集合的"天"D，附带返回当天的全部满足下列条件的条目：
  is_verified === false
```

返回结构示意：

```json
{
  "days": [
    {
      "date": "2026-05-12",
      "transactions": [
        {
          "id": "...",
          "ai_needs_review": true,
          "is_verified": false,
          "ai_confidence": "low",
          /* 其余 transaction 字段同读模型 */
        },
        {
          "id": "...",
          "ai_needs_review": false,
          "is_verified": false,
          "ai_confidence": "high",
          /* 其余 transaction 字段同读模型 */
        }
      ]
    }
  ]
}
```

**两类条目的语义**：

- **待审计条目**：`ai_needs_review === true && is_verified === false`，是请教页的核心操作目标
- **上下文条目**：同一天内 `ai_needs_review === false && is_verified === false` 的其他条目；不是审计目标，但作为消费上下文与待审计条目一同呈现（用户审计单条时需要当天的其他交易作为语境）

**为什么要返回上下文条目？** 请教页采用"按天聚合"的视图组织（与首页日卡片完全一致），保证用户审计某条交易时能看到当天的全部消费上下文。如果只返回待审计条目，天卡片可能只剩零星一两条，失去"一天"的语义价值。上下文条目的可操作性由 §2.3-D filter 决定，与接口返回无关。

**已经 `is_verified === true` 的条目**：不返回（无论 ai_needs_review 取值）；用户已确认/修改过即视为已出列，请教页不再展示。

#### B. 排序规则（v1.1 §2.3-B 重写）

原 v1.1 §2.3-B 的条目级排序（`confidence → amount → time`）**废弃**，改为天级 + 天内两级排序：

**外层（天级排序）**：按当天所有待审计条目（`ai_needs_review = true && is_verified = false`）的 confidence 不定度均值降序排列。

```text
score(t):
  - confidence = "low"    → 3
  - confidence = "medium" → 2
  - confidence = "high"   → 1

day_uncertainty_score(D) =
  avg(score(t) for t in D.transactions
      where ai_needs_review = true && is_verified = false)

天级排序键：day_uncertainty_score 降序
同分时按日期降序（最近的天优先）
```

**注意**：天级排序只看待审计条目的不定度均值，**不看上下文条目**——上下文条目的 confidence 不参与天的排序。

**内层（天内条目排序）**：保留原 v1.1 §2.3-B 的条目级三级排序（`confidence → amount → time`），但作用域是**天内的全部 is_verified = false 条目**（含上下文条目）。

不过，**视觉上更自然的呈现是按时间顺序穿插**（与首页日卡片一致），由表现层规格决定具体呈现顺序；本节只保证接口返回的天内 transactions 数组按时间升序排序，前端可自行重排。

#### C. 出列机制（v1.1 §2.3-C 扩展）

保留原 v1.1 §2.3-C：用户确认或修改分类 → `is_verified = true` → 自动出列。

**新增**：左滑确认动作的接口语义明确化：

```text
左滑确认 = 等同于"用户在审计中再次选择同一分类"
具体行为：设置 is_verified = true，不修改 user_category，不写 user_note
触发条件：用户对单条条目执行左滑确认手势
适用范围：仅可操作的条目（见 §2.3-D filter 联动规则）；只读条目无此动作
```

**批量确认**：

```text
批量确认 = 对勾选集合批量执行左滑确认逻辑
接口表达：传入一组 transaction id，逐条设置 is_verified = true
勾选集合的构成规则由表现层定义（见 SPEC_Inquiry_Page.md）；
接口层不感知"哪些条目可被勾选"，只负责按传入 id 列表执行批量出列
```

**只读条目（见 §2.3-D）的修改路径**：只读条目在请教页内不响应任何接口动作；若用户通过单击进入交易详情页，详情页中的分类修改/锁定动作走详情页原有接口（不属于本节范畴）。但表现层规格可以约定：用户从详情页返回请教页后，**若该条目状态变化（is_verified、user_category、user_note 任一变化）**，则视为已处理，从请教页视图中淡出。淡出动画属于表现层；本节只声明接口层的状态变化是判定依据。

#### D. 视图过滤语义：filter 联动规则（v1.2 新增）

filter 是一个**"可操作 confidence 阈值"**，由它**同时**决定"哪些条目可操作"与"哪些天可见"。

##### D.1 三档 filter 定义

| 档位 | 阈值描述 | 阈值及以下（可操作） | 阈值以上（只读） |
|------|---------|---------------------|----------------|
| `all` | 全部 | high / medium / low 全部可操作 | —— |
| `medium`（默认） | 证据有限及以下 | medium / low 可操作 | high |
| `low` | 仅证据不足 | low 可操作 | medium / high |

##### D.2 两条优先级不变量

**不变量 1**：`ai_needs_review === true` 的条目在任何 filter 下都可操作。即便 confidence = high，只要 `ai_needs_review = true`（因为同商户历史分类不一致、counterparty 首次出现等外部冲突信号入队），就保留可操作状态。这类条目是 AI 自信但系统不放心，属于队列核心。

**不变量 2**：`ai_needs_review === false` 的条目权限完全由 filter 决定：

- confidence 落在 filter 阈值及以下 → 可操作
- confidence 落在 filter 阈值以上 → 只读

##### D.3 可操作性判定（接口侧建议派生）

```text
isOperable(t, filter) =
  t.ai_needs_review === true
  ||
  (
    t.ai_needs_review === false
    && confidence_rank(t.ai_confidence) >= filter_rank(filter)
  )

其中：
  confidence_rank("low")    = 3
  confidence_rank("medium") = 2
  confidence_rank("high")   = 1

  filter_rank("low")    = 3   （仅 low 可操作）
  filter_rank("medium") = 2   （medium 及以下可操作）
  filter_rank("all")    = 1   （全部可操作）
```

接口可不直接返回 `isOperable` 字段，由前端根据 filter 状态实时派生。

##### D.4 天卡片可见性

天 D 在 filter F 下可见，当且仅当：

```text
exists t in D.transactions: isOperable(t, F) === true
```

即：当天至少有一个可操作条目时该天显示；否则该天整体不显示。

天卡片不可见时，其内的所有条目（包括只读条目）均不出现在请教页视图中。

##### D.5 视觉权限边界（接口层声明，由表现层执行）

可操作条目：上 confidence 对应背景色（low → 淡红、medium → 黄、high → 青；具体色值由 `SPEC_Inquiry_Page.md` 与 `SURFACE_SYSTEM.md` 定义）。

只读条目：

- 整体灰显，外覆灰色蒙版（视觉语言与首页"已锁定 / 不可拖拽"条目对齐）
- 不响应左滑手势（无左滑动画）
- 不响应拖拽手势
- 在批量选择视图中**不出现勾选框**，无法被批量勾选
- 单击行为：进入交易详情页（可在详情页修改；返回后按 §2.3-C 处理）

##### D.6 filter 切换的视图行为

filter 切换时触发 §2.3-E 中定义的快照重算。新可见集合与旧可见集合的差集采用平滑过渡动画（具体由表现层定义）。

#### E. 会话视图快照（v1.2 新增）

请教页采用**会话视图快照**机制，避免 AI 引擎运行中的批次产出打断用户审阅。

##### E.1 快照创建

用户进入请教页时，前端按 §A → §D 规则计算一次完整视图，形成**会话视图快照**：

- 实现层：前端 session state（React state / store / 内存对象均可），**不需要落盘文件**
- 内容：天卡片有序数组 + 每天的 transactions 有序数组 + 当前 filter 状态
- 生命周期：与请教页页面 session 绑定；用户离开页面（路由切换 / 退出 App）即销毁

##### E.2 快照内的增量行为

AI 引擎在请教页打开期间跑出新批次产出，导致新条目满足 `ai_needs_review = true` 时：

- **新条目所在天已在快照内** → 追加到该天卡片末尾（视觉上以浮现动画呈现，具体由表现层定义）
- **新条目所在天不在快照内** → 作为新的天卡片整体追加到快照末尾

**关键约束**：任何新增**不刷新已有条目的位置**，**不改变已有天卡片的排序**。已有视图保持稳定，新增内容向后追加。

用户对快照内条目的操作（确认 / 修改 / 批量）**直接更新快照内对应条目状态**，不触发整体重算。

##### E.3 快照重算触发条件

以下事件触发快照完整重算：

- 用户离开请教页再进入
- 用户切换 filter（§2.3-D.6）
- 用户切换账本
- 应用从后台恢复且距上次离开请教页 ≥ 一个由表现层定义的阈值（建议 ≥ 5 分钟，最终值在 `SPEC_Inquiry_Page.md` 定）

##### E.4 快照与持久化的关系

请教页**不引入新的持久化文件**。所有"看上去像持久化"的行为（关闭 App 后重新打开仍能继续审阅未处理条目）都通过快照重算实现——重算时按当前 transaction 真实状态读取，未处理的条目自然仍在队列中，视图按规则重新形成。

唯一的不变量是：**已经 `is_verified = true` 的条目不会再出现在视图里**——这是接口返回过滤逻辑保证的，不依赖任何会话状态。

#### F. 空状态分类（v1.2 新增）

接口层需要让表现层区分以下五种状态（用户感知层文案 / 视觉 / 跳转动作由 `SPEC_Inquiry_Page.md` 定）：

| 状态码 | 接口判定 | 语义 |
|------|---------|------|
| `NO_BILLS` | 当前账本未导入任何账单 | 真空：引导用户去记账页导入 |
| `NO_REVIEW_YET` | 已导入账单，但 AI 引擎从未产出过 `ai_needs_review = true` 的条目（包括"AI 还未运行"和"AI 已运行但全部 high confidence"两种情况；接口层不区分这两子情况） | 真空：可能 AI 还未开始 / 已开始但都很确信 |
| `RUNNING_NON_EMPTY` | AI 引擎正在运行，且队列已包含至少一个 `ai_needs_review = true` 条目 | **不是空状态**；列入分类是为了让表现层在该状态下显示"AI 正在运行，新条目会追加"提示 |
| `ALL_REVIEWED` | 当前 filter 下队列存在过，但用户已完成所有可操作条目（视图可见集合为空），且放宽 filter 后亦无可操作条目 | "教得好"空：给正向反馈 |
| `FILTER_EMPTY` | 当前 filter 下可见集合为空，但放宽 filter 后非空 | 局部空：提示用户放宽 filter |

注意 `NO_REVIEW_YET` 与 `RUNNING_NON_EMPTY` 的关键区别：前者队列从未非空过，后者队列已非空且仍在追加。两者在接口层判定上的依据是"截至当前是否有 `ai_needs_review = true` 条目存在"——`RUNNING_NON_EMPTY` 要求至少有一条。

#### G. UI 形态归属

请教页（Inquiry Page）的具体表现层规格由 `docs/design/spec/SPEC_Inquiry_Page.md` 承接。本节只冻结接口与语义，不涉及视觉 / 动画 / 具体交互。

`SPEC_Inquiry_Page.md` 中定义但**不再返回**到本节修改的内容：

- 三档 confidence 背景色的具体色值
- 左滑动画的物理参数（阈值、回弹、时长、曲线）

- 批量选择视图的交互细节（勾选框样式、批量确认按钮、退场动画时序）
- 空状态各自的文案、视觉、行动按钮
- 单条请教项的信息密度与排版

本节中所有"由表现层定义"的内容均指向上述规格文档。

## 3. 推迟项(不做)

### 3.1 分类后检查会话（`自学习系统改进变更_v0.3.md` §3.2）

**推迟理由**：当前 schema 没有 `confidence` 时，分类后检查会话有"二次审视"的价值。§2.1 落地后，模型自报的 `confidence + uncertaintyReason` 已经覆盖了这个会话能产出的大部分信号，再开一次 LLM call 是 token + 延迟翻倍换边际收益。

**重新评估的触发条件**：跑 ≥ 2 轮真实数据后，如果存在"模型 confidence=high 但实际错误率高"的样本类型，再考虑开此会话。

### 3.2 记忆系统结构化重构（合并：`自学习系统改进变更_v0.3.md` §3.4 + §3.7 + 学习变更可解释性 + 样本溯源）

**合并背景**：本轮原稿中曾包含"学习会话每条 operation 附带 comment + 全局 diagnostic"作为部分覆盖样本溯源诉求的过渡步骤。评估后认为：

- 现有记忆是纯文本 numbered list，加 comment / 元数据需要快照索引与正文分离
- 半步改造会引入临时数据结构，下一步实体/规律分层时还要重写
- 不如一次到位

**统一推迟到第三轮处理**的范围：

- 实体记忆 / 规律记忆结构化分层（`自学习系统改进变更_v0.3.md` §3.7）
- 学习会话每条 operation 的 comment / 全局 diagnostic / 空 operations 解释
- 每条记忆的样本溯源（`自学习系统改进变更_v0.3.md` §3.4，example_id / userNote 来源 / 弱证据标记等）
- 记忆 schema、学习输出格式、快照索引结构联动升级

**留作下一轮专项**：记忆系统结构化重构。

### 3.3 普通 / 推理模型路由（`自学习系统改进变更_v0.3.md` §3.6）

**推迟理由**：路由的关键是"在 LLM 之前如何识别简单样本"。本轮的 `ai_evidence_type` 派生字段与 §2.2 的 `total_score` 各子项是路由数据的雏形，但需要 ≥ 5 轮真实数据积累后才能验证它们与"用便宜模型也不出错"的相关性。本轮做路由是拍脑袋。

### 3.4 审计队列 UI 实现（`自学习系统改进变更_v0.3.md` §3.5）

**v1.2 调整**：本节原 v1.1 表述为"§2.3 已冻结数据接口，UI 设计与交互形态留给下一轮 UI 专项处理"。v1.2 已完成接口语义扩展（§2.3-D / E / F），同步完成请教页表现层规格 `docs/design/spec/SPEC_Inquiry_Page.md`。

**实现工作**仍属于下一轮 release 范畴（接入新接口、实现表现层组件、动画、空状态），不在本轮的"接口与规格冻结"目标之内。

### 3.5 embedding 检索路线

**推迟理由**：本轮 §2.2 用本地传统方法解决 agent 三份报告披露的全部反例。后续若 agent 数据显示存在传统方法无法覆盖的真正语义近义场景（如"快餐" vs "盒饭"等同义词），再单独评估 embedding 引入。本轮不预设此判断。
