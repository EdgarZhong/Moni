# Prototype-Driven Development Workflow v3

本文档定义原型驱动开发的工作流规范，为增量迭代阶段的产品开发提供一套管理 design scope 的机制，确保每个表现层变更从设计探索到工程实现的全链路对齐与可追溯。

## 适用场景

本工作流针对以下协作模式设计：人负责产品体验决策与需求设计，AI 负责 UI/UX 详细设计与正式编码实现。人是设计意图的唯一来源和最终审批者，AI 是高速执行者。

这种协作模式具有与传统多人团队截然不同的信息流转结构。传统团队中，知识分布式存储在团队成员的持续记忆中——设计师脑子里有视觉系统的全貌，工程师脑子里有组件架构的全貌——他们通过日常沟通持续同步，即使文档不完善，人脑中的隐性知识也能补偿。而在人与 AI 的协作中，AI 没有持续记忆。每次会话开始时，AI 的全部认知来源就是它被喂入的文档和上下文。这意味着传统团队里靠"人脑里记着"来维持的一切——这个组件之前为什么这样设计、那个交互上次为什么被否决、这条规格和那条规格之间有什么关联——如果没有被文档化，就等于不存在。

随着产品从 0→1 阶段进入增量迭代，表现层变更的粒度变小、数量变多。每一个变更——无论是新功能、交互调整还是现有状态的修订——都构成一个独立的 design scope，有自己的生命周期：从探索到确认到实现到验收。当 design scope 的数量上升之后，上述协作模式会暴露出四个结构性问题。

## 这套工作流要解决什么问题

### 问题一：设计决策的存活完全依赖文档化

人拍板了一个交互方案，如果它只存在于上一次对话的上下文里而没有沉淀到持久化的载体中，下一次会话的 AI 就不知道这个决策的存在。它可能做出一个看起来合理、但实际上违背了之前决策的实现。而人作为唯一的产品决策者，如果手上同时管着十几个 design scope，也不一定能即时发现这种偏移。

### 问题二：AI 的高产出速度放大混乱

人类工程师一天产出几百行代码，出了问题影响范围有限。AI 可以在一次会话里产出几千行代码，覆盖多个组件、多个状态路径。如果这些产出的方向有偏差，偏差也被同等放大。人事后手工走查的工作量和产出量成正比，而 AI 的产出量远超人类工程师——手工回归测试迅速变得不可承受。

驱动AI的高速产出能力进行设计探索的过程中，如果每一次试错都以直接修改的形式进入主仓库历史，如被否决的方案、临时组件、临时mock数据、实验性样式覆盖——即使最终回滚，也可能已经与其他改动交织。主仓库稳定性会被探索过程侵蚀。

### 问题三：修正路径不清晰

人发现实现与设计不一致，告诉 AI 去改。但 AI 不知道这个不一致的"正确答案"在哪里——是人口头描述的？是规格文档里写的？是原型仓库里跑的？还是人脑子里的最新想法？如果没有一个唯一的、明确的视觉真相源（visual source of truth），每次修正都变成一次重新沟通设计意图的过程，效率极低，而且每次沟通都有信息损耗。

### 问题四：表现层变更的影响边界不可预测

人让 AI 改一个组件的样式，AI 可能连带调整了相邻组件的布局来适配。这个连带调整可能是合理的工程判断，但也可能破坏了另一个已经确认过的 design scope。AI 不知道哪些视觉状态是"已确认不可动"的，除非文档中明确标注了这些边界。

## 工作流如何解决这些问题

本工作流通过三个核心机制应对上述四个问题：

**机制一：用独立原型仓库隔离探索与实现。** 每个 design scope 在原型仓库中完成探索和确认，试错不进入主仓库。这解决了问题二（高速产出的偏差不会污染主仓库）和问题四（每个已确认的 design scope 在原型仓库中有明确的、可视的锁定状态，AI 在实现其他 scope 时不会无意触碰）。此外还能将设计阶段不必要的工程实现细节隔离在上下文之外，最大限度发挥AI的审美与设计能力。

**机制二：用技术栈同构和 mock 层隔离消除迁移翻译层。** 原型代码与主仓库代码使用完全一致的技术栈，迁移操作是搬运加接线，不是翻译加重构。这确保了问题一中设计决策的存活载体（原型代码本身）可以无损地转化为实现，中间不存在引入隐性偏差的翻译环节。同时，原型仓库本身就是问题三中所需的唯一视觉真相源——修正时不需要口头沟通意图，以原型为准。

**机制三：用单向流动纪律保证可追溯性。** 规格从原型仓库单向流入主仓库，任何实现与原型的偏离必须显式标注并经产品确认。这直接回应了问题三（修正路径清晰：发现偏差 → 回到原型仓库修正 → 重新确认 → 再次迁移）和问题四（影响边界可追溯：每个 design scope 的确认状态在原型仓库中有据可查）。

## 核心原则

原型仓库是 UI 表现层的唯一变更入口，主仓库是下游消费者。规格单向流动：原型仓库 → 主仓库，不存在反向路径。

任何表现层变动——无论新页面、交互调整还是状态反馈修改——都先在原型仓库中完成探索和确认，经产品确认后以规格形式流入主仓库实现。

```
原型仓库（探索 → 审查 → 确认）→ 设计规格 → 主仓库（实现）
```

## 一、原型仓库定位

原型仓库是独立于主仓库的设计探索空间。它的职能：

1. 承载所有 UI/UX 表现层探索，所有试错在这里发生
2. 始终反映产品 UI 的最新设计状态
3. 作为主仓库表现层实现的唯一上游依据

原型仓库不是一次性消耗品。它是表现层的长期 source of truth，随产品持续演进。

## 二、技术栈强对齐

### 为什么必须对齐

原型仓库独立于主仓库，如果不加约束，最自然的倾向是降低技术要求换取探索速度——用 JavaScript 代替 TypeScript 省去类型定义，用 inline style 代替项目约定的样式方案快速出效果。

这种降级在探索阶段确实更快，但它制造了一个隐性成本：迁移时需要"翻译"。翻译过程中，组件接口被重新定义，样式被重新组织，行为可能在重写中发生偏移。最终主仓库中的实现与原型仓库中经过确认的方案之间出现了无法被简单 diff 检测到的差异。

技术栈对齐消除的不是开发效率，而是迁移翻译层。当原型代码与主仓库代码使用同一种语言、同一套约束，迁移就从"翻译 + 重构"降级为"搬运 + 接线"。

### 必须对齐的维度

| 维度 | 要求 |
|------|------|
| 语言 | 与主仓库一致（例：主仓库用 TypeScript，原型不允许用 JavaScript） |
| UI 框架 | 与主仓库一致 |
| 样式方案 | 与主仓库一致（例：主仓库用 Tailwind，原型不允许用 inline style 或独立 CSS） |
| 动效库 | 与主仓库一致，不允许引入原型专属动画库 |
| 文件命名 | 遵循主仓库 convention |
| 组件拆分粒度 | 遵循主仓库 convention |

### 禁止事项

- 不允许使用 `any` 类型绕过接口定义——组件的 props interface 在原型中就应该是生产级的
- 不允许引入主仓库中不存在的第三方依赖来实现表现层效果
- 不允许用技术栈降级换取原型开发速度

## 三、Mock 数据层隔离

### 为什么需要显式 mock 层

原型需要数据来驱动 UI 状态，但不应该接入真实业务逻辑。如果数据直接硬编码在组件内部，迁移时就需要逐个组件排查和替换数据来源。如果 mock 函数的签名与真实 service 不一致，迁移时还需要修改组件的调用方式。

显式 mock 层的设计目标是：让 mock 与真实 service 之间的切换成为一次 import 路径替换，而不涉及任何组件代码的修改。这就要求 mock 层的接口签名从一开始就对齐真实 service。

### 结构要求

- 每个原型目录下设独立的 `mock/` 或 `fixtures/` 目录
- Mock 函数的签名必须与主仓库对应 service 方法一致或可直接替换
- 迁移时只需将 import 路径从 mock 切换到真实 service，不需要修改组件代码

### 禁止事项

- 不允许在组件内部硬编码 mock 数据——必须通过 mock 层间接引用
- 不允许 mock 层的接口签名与主仓库 service 存在无法直接替换的差异

### 覆盖要求

Mock 数据应覆盖：

- 正常路径（happy path）
- 关键异常路径（例：网络失败、权限不足、数据格式异常）
- 边界状态（例：空数据、超长文本、极端数值）

覆盖异常和边界的原因是：表现层的设计质量在异常路径上最容易暴露问题——空状态没有设计、错误提示文案缺失、极端数据导致布局崩溃。如果原型阶段只验证 happy path，这些问题会在集成后才被发现，修复成本远高于在原型阶段处理。

## 四、原型审查交付标准

### 为什么需要交付标准

原型的价值在审查环节兑现。如果审查者打开原型后需要自行寻找入口、猜测操作路径、分辨哪些是本次需要关注的内容，审查效率会急剧下降，遗漏率会上升。

实践中观察到的典型问题：审查者进入原型后首先看到的是与当前任务无关的历史原型；审查者不知道某个隐藏状态需要特定操作序列才能触发；审查者不清楚原型有意不覆盖的部分，把缺失当成了问题。

### 交付物

原型提交审查时必须附带以下信息：

1. **启动方式**：一条命令可运行，不需要审查者自行配置环境
2. **状态路径清单**：列出所有可操作的状态路径，不允许存在需要审查者自行发现的隐藏路径
3. **范围边界说明**：明确写出本原型覆盖什么、不覆盖什么

### 审查入口要求

- 每个原型必须有独立的直达入口，不允许审查者从索引页自行寻找
- 不相关的历史原型不应出现在当前审查视野中
- 当主仓库不使用 URL 路由时，直达入口可通过 design scope 导航脚手架实现（见第五章）

## 五、Design Scope 导航脚手架（可选启用）

### 问题背景

当主仓库的产品不使用 URL 路由时（例如 Capacitor 移动端应用，页面切换通过组件状态驱动），原型仓库作为同构镜像也不会有 URL 路由。这意味着所有 design scope 共享同一个入口——审查者打开原型后，需要通过操作（点击按钮、切换 tab）才能到达特定 scope 对应的视图。

当 design scope 数量较少时，这不构成问题。但随着 scope 增多，审查体验会退化：审查者不知道怎么到达某个 scope、不确定当前看到的是不是正确的 scope、在探索一个 scope 时意外进入了另一个 scope 的视图。

### 解决方案：开发态导航壳

在原型仓库中引入一个极薄的路由层，作为审查和 scope 管理的专用脚手架。这个路由层的性质是**开发工具**，不是产品代码——类似于 Storybook 的导航面板承载组件 story，但 story 中展示的组件本身是生产级的。

具体实现：原型仓库的入口文件通过 URL hash 或路径参数决定渲染哪个视图组件。每个 design scope 对应一个路由，审查者通过直达 URL 访问特定 scope。

```
原型仓库入口（脚手架层，不迁移）
  ├── /#/import-bill     → 导入账单 scope 的页面组件
  ├── /#/budget           → 预算设置 scope 的页面组件
  └── /#/home             → 首页 scope 的页面组件
```

### 与技术栈强对齐的关系

这个导航壳不违反技术栈强对齐原则，因为它不属于产品代码层。需要严格遵守以下边界：

- 导航壳只存在于原型仓库的入口文件中，只控制"渲染哪个组件"，不参与组件内部的任何逻辑
- 页面组件本身与主仓库完全一致——它们不知道也不依赖导航壳的存在
- 迁移时只搬运页面组件，导航壳留在原型仓库，主仓库的入口文件继续使用其原有的导航方式（状态驱动、tab 切换等）
- 导航壳的代码应集中在一个文件中，并以注释或命名方式明确标记为脚手架（例：`scaffold.tsx` 或 `dev-router.tsx`），防止被误认为产品代码

### 附加能力

导航壳建立之后，可以在不侵入产品组件的前提下扩展审查辅助功能，达成类似storybook的导航结构和生命周期管理能力：

- Scope 索引页：列出所有 design scope 的名称、当前状态（探索中 / 已确认 / 已迁移）、直达链接
- 状态变体切换：对于同一个 scope，通过 URL 参数切换不同的 mock 数据集，直达特定状态（正常 / 空数据 / 错误 / 边界值），省去审查者手工操作触发的步骤
- 视觉走查对照：迁移验收时，在导航壳中并排展示原型组件和主仓库截图，辅助一致性比对

这些扩展都属于脚手架层的能力，不触碰产品组件代码，不影响迁移操作。

### 启用时机

导航脚手架是可选组件，不是工作流的强制要求。建议在以下条件满足时启用：

- Design scope 数量增长到口头描述审查路径已经不够高效的程度
- 审查者需要频繁在不同 scope 之间切换对照
- 状态变体较多，手工触发特定状态的操作成本过高

在 scope 数量较少的阶段，直接使用原型的自然导航（与主仓库一致的操作路径）即可满足审查需求。

## 六、原型内容规范

### 文案

原型页面中出现的所有文案必须是用户可见文案。不允许将内部决策原因、技术说明、占位文字写进 UI 文案。

这条规则来自实践中的一个常见错误：原型开发者把"为了避免挤压某某组件"这类内部理由写成了界面上的提示语。审查者看到的应该是用户会看到的东西，而不是开发者的内部注释。内部决策原因属于 brief 或 commit message，不属于 UI。

### 交互

优先使用目标平台原生能力。例如移动端数字输入应优先使用系统输入法，而不是在页面内自建数字键盘。原因是原型需要验证的是真实用户体验，而自建组件替代平台原生能力会制造与实际不符的虚假交互预期。

### 视觉保真度

原型的视觉输出应达到可直接作为实现参照的保真度。不允许出现"先用简陋 UI 验证逻辑，后续再美化"的做法。

这是原型驱动开发区别于传统快速原型（rapid prototyping）的关键点。传统快速原型的目标是验证交互逻辑，视觉层可以粗糙。但在本工作流中，原型是实现的唯一上游依据——如果原型视觉粗糙，实现者要么照搬粗糙结果，要么自行做视觉决策。前者产出质量不达标，后者意味着实现者越权做了应该在设计阶段完成的决定。

## 七、设计阶段与实现阶段分离

### 为什么需要显式阶段门

实践中观察到的一个高频事故是：原型已经足够高保真，代码甚至可以直接复用，于是探索者在没有获得产品确认的情况下，顺手把设计结论写进了主仓库的正式代码。

这种"顺手提交"的危害在于，它绕过了产品确认环节。一个看起来完成度很高的原型，可能只是探索过程中的某个中间状态，产品还没有确认它是最终方案。如果它已经进入主仓库，后续推翻的成本就从"在原型仓库里删掉"变成了"在主仓库里回滚并处理关联影响"。

### 规则

- 设计阶段的产出是经产品确认的设计规格，不是主仓库中的代码变更
- 未经产品明确授权进入实现阶段前，任何设计结论不允许写入主仓库
- 原型即使已经足够高保真、甚至代码可以直接复用，也不能替代"进入实现"的明确授权

阶段门不是流程官僚，它是防止探索性成本意外转化为主仓库维护成本的断路器。

## 八、迁移纪律

### 为什么迁移需要纪律

技术栈对齐和 mock 层隔离使得迁移操作本身非常简单——搬运组件代码、替换 import 路径。但正因为操作简单，实现者容易在迁移过程中"顺便优化"：调整一下组件拆分、优化一下 prop 命名、重构一下状态管理。

这些优化本身可能是合理的工程决策，但它们打破了原型与实现之间的可追溯性。审查者不再能通过 diff 确认"主仓库中的实现是否忠实反映了原型中确认的方案"。迁移纪律的目标不是禁止优化，而是要求所有偏离原型的改动都是显式的、可审计的。

### 一致性要求

- 组件结构、prop 接口、视觉输出必须与原型保持一致
- 不允许迁移过程中进行静默重构——任何偏离原型的改动必须标注原因并报告产品确认

### 迁移操作

- 将原型组件代码搬入主仓库对应目录
- 将 mock 层 import 替换为真实 service import
- 除上述替换外，不应存在其他代码改动

### 迁移验收

- 迁移完成后，在目标设备上对照原型做视觉走查
- 走查确认视觉输出、交互行为、状态路径与原型一致
- 存在偏差时，以原型为准修正实现，而非反过来修改原型

以原型为准修正实现，而不是反过来，这一点至关重要。如果允许"实现中发现原型有问题就在主仓库中就地修改"，单向流动的原则就被打破了——主仓库开始反向影响设计决策，原型仓库作为 source of truth 的地位就名存实亡。

## 九、原型仓库的持续演进

### 为什么不能归档

如果原型在迁移完成后被归档或废弃，后续的表现层修改就只能直接在主仓库中进行——这意味着回到了"在主仓库中探索 UI"的起点，本文档解决的所有问题会重新出现。

原型仓库是长期维护对象，因为它承担的不是"某次功能开发的设计辅助"，而是"产品表现层的持续设计管理"。

### 维护节奏

- 新功能：先在原型仓库中完成完整探索和确认
- 现有页面修改：先在原型仓库中更新对应原型，确认后再流入主仓库
- 原型仓库应始终反映产品 UI 的最新确认状态

### 与主仓库的同步边界

规格单向流动意味着：

- 原型仓库不需要跟踪主仓库的非表现层变更（后端逻辑、数据结构等）
- 主仓库的表现层实现始终以原型仓库的最新确认状态为准
- 如果主仓库实现过程中发现原型设计存在问题，应回到原型仓库修正，而非在主仓库中就地修改

### 仓库协作边界

原型仓库与主仓库必须是两个独立运行环境。原型仓库是信息参考源和表现层 source of truth，主仓库是正式产品运行时代码来源，二者之间禁止建立运行时代码耦合。

必须遵守：

- 主仓库不得通过 `import`、`require`、Vite alias、TypeScript paths、npm workspace、npm link、submodule 指针、构建脚本或跨仓库相对路径引用原型仓库代码
- 原型仓库不得通过同类方式引用主仓库运行时代码
- 两个仓库必须各自独立安装依赖、独立启动、独立构建、独立通过验证
- 允许复制代码或规格；复制后代码归目标仓库所有，并由目标仓库自己的依赖、mock 或真实 service 接线
- 原型仓库的 mock / fixtures 不迁移到主仓库正式运行时；迁移时只替换为主仓库真实 service import

## 十、适用边界

本工作流并非万能方案。以下是它的最佳适用场景和已知局限。

### 最佳适用

- 面向用户的产品，UI/UX 质量是核心竞争力
- 小团队（1-3 人），产品决策者直接参与设计审查闭环
- 表现层与业务逻辑可以干净分离
- 变更节奏是"每个功能认真打磨"而非"高频实验快速迭代"

### 已知局限

- **团队规模扩大时**：多人同时写入原型仓库会产生合并冲突和审查瓶颈，单一 source of truth 在多写入者场景下会退化。规模化团队通常用 Design System + Component Library（如 Storybook）以组件契约替代原型代码作为 source of truth
- **高频实验场景**：A/B 测试密集的增长阶段，每次表现层变更都走原型往返是纯开销。此时 Feature Flag 机制更合适——让未确认方案安全地存在于主仓库但不对用户可见，用生产数据做确认
- **设计系统成熟后**：当组件库足够完备，新功能是现有组件的排列组合时，原型的视觉验证价值大幅下降，文字规格 + 组件名引用即可替代
- **表现层与业务逻辑不可分离时**：数据密集型仪表盘、实时协作产品等场景中，mock 数据层无法忠实模拟真实行为，原型给出的体验预期可能产生误导
